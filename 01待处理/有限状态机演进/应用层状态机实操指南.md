# 应用层状态机实操指南

> [!NOTE]
> **适用场景**:用户交互逻辑(菜单、工作模式切换)、设备工作流(煮咖啡、洗衣服)、协议状态机(连接/认证/通信)
> **核心关注**:业务逻辑正确性、用户体验流畅性、状态可观测性
> **前置阅读**:[[有限状态机基础入门#1. 为什么需要状态机？|基础入门]]、[[有限状态机演进详解#1. 全局演进总览|演进详解]]

---

## 1. 核心思想:业务流程 → 状态图

应用层状态机的本质:**把业务流程的每个"阶段"转化为一个状态,把用户操作和系统事件转化为触发条件**。

```
[业务流程的每个阶段] = [状态机中的一个状态]
[用户操作/系统事件]  = [状态机中的事件]
[阶段间的流转规则]   = [状态机的转移条件]
```

**应用层 vs 驱动层的关键区别**(对照 [[驱动层状态机实操指南]]):

| 维度 | 驱动层 | 应用层 |
|---|---|---|
| 状态来源 | 硬件时序图 | 业务流程图/需求文档 |
| 事件来源 | 中断/超时/电平跳变 | 按键/传感器数据/定时器/其他模块通知 |
| 时间约束 | 硬实时(us 级) | 软实时(ms~s 级) |
| 错误处理 | 超时重试/放弃 | 用户提示/安全降级/自动恢复 |
| 状态可见性 | 内部调试 | 需要 UI 展示(当前模式指示灯/屏幕) |
| 状态数 | 通常 5~10 | 通常 3~7,但可能嵌套 |

## 2. 版本一:逻辑构建(Switch-Case 映射)

### 2.1 适用场景

- 业务流程简单,线性流转
- 状态数 ≤ 5,无嵌套
- 快速原型

### 2.2 原理讲解:业务流程 → 状态枚举

以"咖啡机工作流"为例:

```
待机 → 按下制作键 → 加热 → 温度达标 → 萃取 → 计时完成 → 完成 → 自动回待机
```

**形象比喻**:像**工厂流水线**,原料经过一道道工序,每道工序就是一个状态,完成检验(事件)后流入下一道。

### 2.3 操作步骤清单

- [ ] Step 1:梳理业务流程——画出完整的状态流转图,标注每个转移的条件和动作
- [ ] Step 2:定义状态枚举——每个业务阶段一个枚举值,必须包含 IDLE
- [ ] Step 3:定义事件枚举——用户操作 + 系统事件(超时/传感器通知等)
- [ ] Step 4:实现状态机主函数——switch(current_state) + switch(event) 嵌套
- [ ] Step 5:定义动作函数——状态切换时需要执行的操作(开加热器/启动泵/亮灯)
- [ ] Step 6:实现心跳驱动——定时器扫描事件源,调用状态机

### 2.4 代码模板

```c
/* Step 2: 状态定义 */
typedef enum {
    APP_STATE_IDLE,      /* 待机 */
    APP_STATE_HEATING,   /* 加热中 */
    APP_STATE_BREWING,   /* 萃取中 */
    APP_STATE_DONE       /* 制作完成 */
} app_state_t;

/* Step 3: 事件定义 */
typedef enum {
    APP_EVT_NONE,
    APP_EVT_START,          /* 按下制作键 */
    APP_EVT_TEMP_READY,     /* 温度达标 */
    APP_EVT_BREW_DONE,      /* 萃取计时完成 */
    APP_EVT_TIMEOUT,        /* 超时 */
    APP_EVT_CANCEL          /* 取消 */
} app_event_t;

/* Step 5: 动作函数 */
static void Action_HeatOn(void)   { /* 开加热器 */ }
static void Action_HeatOff(void)  { /* 关加热器 */ }
static void Action_PumpOn(void)   { /* 开泵 */ }
static void Action_PumpOff(void)  { /* 关泵 */ }
static void Action_LedBlink(void) { /* LED 闪烁 */ }
static void Action_LedOn(void)    { /* LED 常亮 */ }

/* Step 4: 状态机主函数 */
static app_state_t g_state = APP_STATE_IDLE;

void App_FsmHandler(app_event_t event)
{
    switch (g_state)
    {
    case APP_STATE_IDLE:
        if (event == APP_EVT_START)
        {
            Action_HeatOn();
            Action_LedBlink();
            g_state = APP_STATE_HEATING;
        }
        break;

    case APP_STATE_HEATING:
        if (event == APP_EVT_TEMP_READY)
        {
            Action_PumpOn();
            g_state = APP_STATE_BREWING;
        }
        else if (event == APP_EVT_CANCEL || event == APP_EVT_TIMEOUT)
        {
            Action_HeatOff();
            Action_LedOn();
            g_state = APP_STATE_IDLE;
        }
        break;

    case APP_STATE_BREWING:
        if (event == APP_EVT_BREW_DONE)
        {
            Action_PumpOff();
            Action_HeatOff();
            Action_LedOn();
            g_state = APP_STATE_DONE;
        }
        else if (event == APP_EVT_CANCEL)
        {
            Action_PumpOff();
            Action_HeatOff();
            g_state = APP_STATE_IDLE;
        }
        break;

    case APP_STATE_DONE:
        if (event == APP_EVT_TIMEOUT)
        {
            g_state = APP_STATE_IDLE;
        }
        break;
    }
}

/* Step 6: 心跳驱动(参考 [[有限状态机基础入门#5. 状态机的心跳|心跳机制]]) */
void App_1msTick(void)
{
    static uint16_t brew_counter = 0;
    static uint16_t done_counter = 0;

    switch (g_state)
    {
    case APP_STATE_BREWING:
        if (++brew_counter >= BREW_TIME_MS)
        {
            brew_counter = 0;
            App_FsmHandler(APP_EVT_BREW_DONE);
        }
        break;
    case APP_STATE_DONE:
        if (++done_counter >= DONE_DISPLAY_MS)
        {
            done_counter = 0;
            App_FsmHandler(APP_EVT_TIMEOUT);
        }
        break;
    default:
        brew_counter = 0;
        done_counter = 0;
        break;
    }
}
```

### 2.5 V1 版的局限性

- **状态与动作混在一起**:切换 HEATING → BREWING 时,关加热器和开泵的动作直接写在 if 分支里,容易遗漏
- **全局变量暴露**:g_state 任何函数都能改,无保护
- **无生命周期管理**:没有"进入加热时开灯,离开加热时关灯"的自动机制
- **取消逻辑重复**:HEATING 和 BREWING 都有取消分支,动作几乎一样却要写两遍

> [!TIP]
> 取消逻辑重复的问题,在 V2 中通过 **EXIT 机制**优雅解决——无论从哪个状态离开,EXIT 自动执行清理。

## 3. 版本二:功能完善(ENTRY/EXIT + 查表法)

### 3.1 适用场景

- 状态切换时需要**资源管理**(开关外设、启停定时器)
- 需要代码复用,减少重复逻辑
- 团队协作,每人负责不同状态函数

### 3.2 原理讲解:ENTRY/EXIT 消除重复逻辑

**形象比喻**:V1 的状态切换像**搬出旧公寓时自己打扫、搬入新公寓时自己布置**。V2 则像**物业管理**——你只需拎包入住,物业自动帮你打扫旧房、布置新房。

**V1 的痛点**:
```c
/* 取消操作:HEATING 和 BREWING 各写一遍,几乎一样 */
case APP_STATE_HEATING:
    if (event == APP_EVT_CANCEL) { Action_HeatOff(); g_state = IDLE; }
    break;
case APP_STATE_BREWING:
    if (event == APP_EVT_CANCEL) { Action_PumpOff(); Action_HeatOff(); g_state = IDLE; }
    break;
```

**V2 的解法**:EXIT 自动清理,取消只需切换状态:
```c
case APP_EVT_CANCEL:
    self->state = APP_STATE_IDLE;
    return STATUS_TRAN;   /* 调度器自动调用 EXIT,执行清理 */
```

### 3.3 原理讲解:查表法替代 switch

将状态枚举值作为数组下标,直接取出对应的处理函数指针。这与 [[有限状态机演进详解#2.3 V3 函数指针数组映射调用（查表法）|查表法]]的思想一致。

**形象比喻**(引用 [[有限状态机基础入门#3.2 函数指针：把"行为"变成"数据"|函数指针]]):
- switch-case = "问路":挨个问"你是状态1吗?"
- 函数指针数组 = "查通讯录":下标直达,O(1)

### 3.4 操作步骤清单

- [ ] Step 1:定义状态处理函数签名——统一为 `status_t handler(fsm_t *self, event_t evt)`
- [ ] Step 2:为每个状态实现处理函数——含 ENTER/EXIT 分支
- [ ] Step 3:建立函数指针数组——用 C99 指定初始化 `[STATE_XXX] = handler_xxx`
- [ ] Step 4:实现调度器——dispatch 函数自动处理 ENTER/EXIT
- [ ] Step 5:统一状态切换宏——`TRAN_TO(target)` 防呆

### 3.5 代码模板

```c
/* Step 1: 统一类型定义 */
typedef enum { STATUS_HANDLED, STATUS_TRAN } status_t;
typedef struct app_fsm app_fsm_t;
typedef status_t (*app_handler_t)(app_fsm_t *self, app_event_t event);

/* 应用状态机结构体 */
struct app_fsm
{
    app_state_t state;
    app_handler_t *handlers;
    uint16_t timer_counter;    /* 通用计时器 */
};

/* Step 2: 各状态处理函数 */
static status_t handle_idle(app_fsm_t *self, app_event_t event)
{
    switch (event)
    {
    case APP_EVT_START:
        return TRAN_TO(APP_STATE_HEATING);
    default:
        return STATUS_HANDLED;
    }
}

static status_t handle_heating(app_fsm_t *self, app_event_t event)
{
    switch (event)
    {
    case EVENT_STATE_ENTER:
        Action_HeatOn();
        Action_LedBlink();
        self->timer_counter = 0;
        return STATUS_HANDLED;

    case EVENT_STATE_EXIT:
        Action_HeatOff();      /* 无论去哪里,EXIT 自动关加热 */
        return STATUS_HANDLED;

    case APP_EVT_TEMP_READY:
        return TRAN_TO(APP_STATE_BREWING);

    case APP_EVT_CANCEL:
        return TRAN_TO(APP_STATE_IDLE);   /* EXIT 自动关加热,无需手动 */

    case APP_EVT_TIMEOUT:
        return TRAN_TO(APP_STATE_IDLE);

    default:
        return STATUS_HANDLED;
    }
}

static status_t handle_brewing(app_fsm_t *self, app_event_t event)
{
    switch (event)
    {
    case EVENT_STATE_ENTER:
        Action_PumpOn();
        self->timer_counter = 0;
        return STATUS_HANDLED;

    case EVENT_STATE_EXIT:
        Action_PumpOff();      /* 无论去哪里,EXIT 自动关泵 */
        return STATUS_HANDLED;

    case APP_EVT_BREW_DONE:
        return TRAN_TO(APP_STATE_DONE);

    case APP_EVT_CANCEL:
        return TRAN_TO(APP_STATE_IDLE);   /* EXIT 自动关泵 */

    default:
        return STATUS_HANDLED;
    }
}

/* Step 3: 函数指针数组(C99 指定初始化) */
static app_handler_t app_handlers[] = {
    [APP_STATE_IDLE]    = handle_idle,
    [APP_STATE_HEATING] = handle_heating,
    [APP_STATE_BREWING] = handle_brewing,
    [APP_STATE_DONE]    = handle_done
};

/* Step 4: 调度器 */
void App_Dispatch(app_fsm_t *self, app_event_t event)
{
    app_state_t prev = self->state;
    status_t s = self->handlers[self->state](self, event);

    if (s == STATUS_TRAN)
    {
        self->handlers[prev](self, EVENT_STATE_EXIT);
        self->handlers[self->state](self, EVENT_STATE_ENTER);
    }
}

/* Step 5: TRAN_TO 宏(参考 [[有限状态机演进详解#5.1 TRAN_TO 宏的设计精妙|TRAN_TO 宏设计]]) */
#define TRAN_TO(target) (self->state = (target), STATUS_TRAN)
```

### 3.6 V2 的改进与遗留问题

**改进**:
- EXIT 消除取消逻辑重复:无论从 HEATING 或 BREWING 取消,EXIT 自动关对应设备
- 查表法:新增状态只需加一个函数和一行数组初始化,无需改调度器
- 结构体封装:为多实例留出扩展空间

**遗留**:
- 如何让 UI 层知道当前状态?(如 OLED 显示"加热中")
- 如何处理状态机的嵌套?(如"制作中"包含"加热""萃取"子状态)
- 状态机之间的协作如何实现?(如咖啡机状态机需要等待水位传感器状态机完成)

## 4. 版本三:系统集成(多状态机协作 + 事件通知)

### 4.1 适用场景

- 多个状态机需要协同工作
- 需要 UI 层实时反映状态
- RTOS 多任务通信

### 4.2 原理讲解:多状态机协作

**问题**:咖啡机需要同时管理"制作流程"和"水位监控",两个状态机相互独立但需要通信。

**解决方案**:事件驱动 + 回调/队列通知。

```
┌──────────────┐   EVT_WATER_LOW   ┌──────────────┐
│  水位状态机   │ ─────────────────→ │  制作状态机   │
│ (独立运行)   │                    │ (独立运行)   │
└──────────────┘                    └──────┬───────┘
                                           │ EVT_STATE_CHANGED
                                           ▼
                                    ┌──────────────┐
                                    │   UI 展示层   │
                                    └──────────────┘
```

**形象比喻**:像**公司部门协作**——研发部(水位状态机)做完工作发邮件(事件)通知产品部(制作状态机),产品部再通知设计部(UI 层)。各部门独立运转,通过消息通信。

### 4.3 原理讲解:状态变化通知

应用层状态机需要让外部(特别是 UI 层)知道当前状态。三种通知方式:

| 方式 | 实时性 | 耦合度 | 适用场景 |
|---|---|---|---|
| 轮询读取 state | 低 | 低 | 简单指示灯 |
| 状态变化回调 | 高 | 中 | OLED 实时刷新 |
| 事件广播 | 高 | 低 | 多订阅者(日志+UI+云端) |

### 4.4 操作步骤清单

- [ ] Step 1:定义状态机间通信事件——跨状态机的事件枚举
- [ ] Step 2:定义状态变化回调类型——`typedef void (*on_state_changed_t)(app_state_t old, app_state_t new)`
- [ ] Step 3:在调度器中触发通知——状态切换时调用回调或发送事件
- [ ] Step 4:实现订阅/注册机制——UI 层注册回调,日志模块也注册回调
- [ ] Step 5:RTOS 环境下用队列替代回调——多任务间安全通信

### 4.5 代码模板

```c
/* Step 2: 状态变化回调类型 */
typedef void (*on_state_changed_t)(app_state_t old_state, app_state_t new_state);

/* 增强的应用状态机结构体 */
struct app_fsm
{
    app_state_t state;
    app_handler_t *handlers;
    uint16_t timer_counter;

    /* 通知相关 */
    on_state_changed_t on_state_changed;
    void *user_data;

    /* 状态机 ID(用于日志和调试) */
    const char *name;
};

/* Step 3: 修改调度器,在状态切换时触发通知 */
void App_Dispatch(app_fsm_t *self, app_event_t event)
{
    app_state_t prev = self->state;
    status_t s = self->handlers[self->state](self, event);

    if (s == STATUS_TRAN)
    {
        self->handlers[prev](self, EVENT_STATE_EXIT);
        self->handlers[self->state](self, EVENT_STATE_ENTER);

        /* Step 3: 通知订阅者 */
        if (self->on_state_changed)
        {
            self->on_state_changed(prev, self->state);
        }
    }
}

/* Step 4: UI 层注册回调 */
void ui_on_state_changed(app_state_t old_state, app_state_t new_state)
{
    switch (new_state)
    {
    case APP_STATE_IDLE:    OLED_ShowString("待机");    break;
    case APP_STATE_HEATING: OLED_ShowString("加热中"); break;
    case APP_STATE_BREWING: OLED_ShowString("萃取中"); break;
    case APP_STATE_DONE:    OLED_ShowString("完成!");   break;
    }
}

/* 初始化时注册 */
app_fsm_t coffee_fsm;
coffee_fsm.on_state_changed = ui_on_state_changed;
```

### 4.6 多状态机协作示例

```c
/* 水位监控状态机 */
typedef enum { WATER_OK, WATER_LOW, WATER_REFILLING } water_state_t;

static app_fsm_t water_fsm;
static app_fsm_t coffee_fsm;

/* 水位状态机检测到缺水时,通知咖啡机 */
static status_t handle_water_ok(app_fsm_t *self, app_event_t event)
{
    switch (event)
    {
    case APP_EVT_WATER_LOW:
        return TRAN_TO(WATER_LOW);
    }
}

static status_t handle_water_low(app_fsm_t *self, app_event_t event)
{
    switch (event)
    {
    case EVENT_STATE_ENTER:
        /* 通知咖啡机暂停 */
        App_Dispatch(&coffee_fsm, APP_EVT_WATER_LOW);
        /* 启动加水 */
        Valve_Open();
        return STATUS_HANDLED;
    case EVENT_STATE_EXIT:
        Valve_Close();
        return STATUS_HANDLED;
    case APP_EVT_WATER_FULL:
        /* 通知咖啡机恢复 */
        App_Dispatch(&coffee_fsm, APP_EVT_WATER_OK);
        return TRAN_TO(WATER_OK);
    }
}
```

## 5. 应用层特有的设计模式

### 5.1 超时自动回退

应用层状态机常见需求:某个状态停留过久,自动回退到安全状态。

```c
static status_t handle_heating(app_fsm_t *self, app_event_t event)
{
    switch (event)
    {
    case EVENT_STATE_ENTER:
        self->timer_counter = 0;
        return STATUS_HANDLED;

    case APP_EVT_TICK:   /* 心跳 Tick 事件 */
        if (++self->timer_counter > HEATING_TIMEOUT_MS)
        {
            return TRAN_TO(APP_STATE_IDLE);   /* 超时自动回退 */
        }
        return STATUS_HANDLED;
    }
}
```

**与驱动层超时的区别**:
- 驱动层超时 = 硬件故障保护(必须硬实时)
- 应用层超时 = 用户体验优化(软实时,甚至可给用户提示后再回退)

### 5.2 状态守卫(Guard Condition)

某些转移需要额外条件,不仅仅是"收到事件就切换":

```c
case APP_EVT_START:
    /* 守卫条件:水位不足时不允许启动 */
    if (Water_GetLevel() < LEVEL_MIN)
    {
        OLED_ShowString("请先加水!");
        return STATUS_HANDLED;   /* 事件消费但不切换 */
    }
    return TRAN_TO(APP_STATE_HEATING);
```

**形象比喻**:守卫就像**门禁**——你有通行证(事件),但门禁还要检查你是不是本楼住户(条件)。

### 5.3 状态嵌套(层次状态机 HSM 概念入门)

当某些状态共享行为时,可以设计"父状态":

```
制作中(父状态:共享"可取消""可暂停")
  ├── 加热(子状态)
  └── 萃取(子状态)
```

**简单实现**:在子状态处理函数中,未识别的事件"冒泡"给父状态:

```c
static status_t handle_heating(app_fsm_t *self, app_event_t event)
{
    switch (event)
    {
    case APP_EVT_TEMP_READY:
        return TRAN_TO(APP_STATE_BREWING);
    default:
        /* 未处理的事件交给父状态(制作中) */
        return handle_making(self, event);
    }
}

/* 父状态:处理所有子状态共享的事件 */
static status_t handle_making(app_fsm_t *self, app_event_t event)
{
    switch (event)
    {
    case APP_EVT_CANCEL:
        return TRAN_TO(APP_STATE_IDLE);   /* 子状态通用取消 */
    case APP_EVT_PAUSE:
        return TRAN_TO(APP_STATE_PAUSED); /* 子状态通用暂停 */
    default:
        return STATUS_HANDLED;
    }
}
```

> [!NOTE]
> 这是层次状态机(HSM)的极简实现。完整的 HSM 框架(如 QP)会自动处理事件冒泡,但裸机项目用上述"手动冒泡"已足够。

## 6. 调试技巧:状态可观测性

### 6.1 状态名称字符串表

调试时最痛苦的是看到 `current_state = 3`,还要查枚举定义。解决方案:

```c
/* 与状态枚举一一对应的名称表 */
static const char *state_name[] = {
    [APP_STATE_IDLE]    = "IDLE",
    [APP_STATE_HEATING] = "HEATING",
    [APP_STATE_BREWING] = "BREWING",
    [APP_STATE_DONE]    = "DONE"
};

/* 调试输出 */
printf("[%s] %s -> %s\r\n",
    self->name,
    state_name[prev],
    state_name[self->state]);
```

### 6.2 状态日志缓冲区

在定时器中断中 printf 会阻塞,用环形缓冲区记录,主循环中输出:

```c
#define LOG_SIZE 32
typedef struct {
    uint32_t timestamp;
    app_state_t old_state;
    app_state_t new_state;
    app_event_t event;
} state_log_t;

static state_log_t g_log[LOG_SIZE];
static uint8_t g_log_head = 0;

/* 在状态切换时记录(极速,不加锁) */
void Log_StateChange(app_state_t old_s, app_state_t new_s, app_event_t evt)
{
    g_log[g_log_head].timestamp = Timer_GetMs();
    g_log[g_log_head].old_state = old_s;
    g_log[g_log_head].new_state = new_s;
    g_log[g_log_head].event = evt;
    g_log_head = (g_log_head + 1) % LOG_SIZE;
}

/* 主循环中慢慢输出 */
void Log_PrintAll(void)
{
    /* ... 遍历缓冲区,printf 输出 ... */
}
```

### 6.3 状态指示 LED

最廉价的调试手段:不同状态点亮不同颜色的 LED:

```c
void ui_on_state_changed(app_state_t old, app_state_t new)
{
    switch (new)
    {
    case APP_STATE_IDLE:    LED_SetColor(LED_OFF);     break;
    case APP_STATE_HEATING: LED_SetColor(LED_RED);     break;
    case APP_STATE_BREWING: LED_SetColor(LED_GREEN);   break;
    case APP_STATE_DONE:    LED_SetColor(LED_BLUE);    break;
    }
}
```

## 7. 常见陷阱与最佳实践

> [!WARNING]
> **陷阱 1:事件被"吞掉"**
> 在状态处理函数中收到一个不相关的事件,如果没有 default 分支返回 STATUS_HANDLED,事件可能被静默丢弃。**每个状态函数都必须有 default: return STATUS_HANDLED**。

> [!WARNING]
> **陷阱 2:状态机之间的循环通知**
> 状态机 A 的状态切换通知了状态机 B,B 处理后又通知 A,形成无限递归。**设计时必须明确事件流向,避免双向依赖**。

> [!CAUTION]
> **陷阱 3:心跳中直接调用耗时操作**
> 心跳回调中执行 OLED 刷新、字符串格式化等操作,会导致其他心跳周期抖动。**耗时操作应设置标志位,主循环中执行**。

> [!CAUTION]
> **陷阱 4:忽略 IDLE 状态的设计**
> IDLE 不是"什么都不做",它是**安全的默认状态**。所有超时、错误、取消都应回退到 IDLE。IDLE 的 ENTRY 应当确保所有外设关闭、UI 显示待机画面。

> [!IMPORTANT]
> **最佳实践 1:先画状态转移图再写代码**
> 应用层状态机比驱动层更容易出现"遗漏状态"。状态转移图能帮你发现死路(无法退出的状态)和缺失的取消路径。

> [!IMPORTANT]
> **最佳实践 2:每个状态都有取消/回退路径**
> 用户随时可能按取消键,系统随时可能出故障。每个非 IDLE 状态都必须有至少一条回退路径。

> [!IMPORTANT]
> **最佳实践 3:状态名称表是调试必备**
> 用 C99 指定初始化维护 `[STATE_XXX] = "XXX"` 字符串表,成本极低,调试收益巨大。

> [!IMPORTANT]
> **最佳实践 4:事件枚举预留扩展空间**
> `APP_EVT_USER = 100`,后续扩展事件从 100 开始,避免修改已有枚举值。

## 8. 版本选型指南

| 版本 | 状态数 | 状态机数 | RTOS | 推荐场景 |
|---|---|---|---|---|
| V1 逻辑构建 | ≤5 | 1 | 否 | 简单工作流、快速原型 |
| V2 功能完善 | 5~10 | 1~2 | 否 | 产品级单状态机 |
| V3 系统集成 | 不限 | 多个 | 是/否 | 多状态机协作、UI 联动 |

**选型建议**:
- 简单家电(电饭煲/加湿器)→ V1 或 V2
- 带屏幕设备(咖啡机/洗衣机)→ V2(查表法 + ENTRY/EXIT + 状态通知)
- 多模块协作(物联网网关/智能家居中枢)→ V3

## 9. 总结速查

- **核心映射**:业务流程的每个阶段 = 一个状态,用户操作/系统事件 = 触发条件
- **三版递进**:V1(switch-case 映射)→ V2(ENTRY/EXIT + 查表法)→ V3(多机协作 + 事件通知)
- **ENTRY/EXIT 价值**:消除取消逻辑重复,保证资源安全释放
- **应用层特有**:超时自动回退、状态守卫、状态嵌套(HSM 简化)
- **调试三板斧**:状态名称表、日志缓冲区、指示 LED
- **铁律**:每个非 IDLE 状态都有取消/回退路径
