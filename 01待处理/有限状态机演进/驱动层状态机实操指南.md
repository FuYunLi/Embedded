# 驱动层状态机实操指南

> [!NOTE]
> **适用场景**:硬件时序驱动(DHT11、I2C、SPI、UART 协议、单总线设备等)
> **核心关注**:精确时序、中断响应、非阻塞设计
> **前置阅读**:[[有限状态机基础入门]]、[[有限状态机演进详解]]

---

## 1. 核心思想:时序图 → 状态图

驱动层状态机的本质:**把硬件时序图的每一个"等待点"转化为一个状态**。

```
[时序图中的每一段等待] = [状态机中的一个状态]
[时序图中的电平跳变]   = [状态机中的事件/触发条件]
```

**驱动层状态机的独特性**:

| 维度 | 驱动层 | 应用层 |
|---|---|---|
| 状态粒度 | 细(us 级时序节点) | 粗(用户可感知阶段) |
| 事件来源 | 硬件中断(边沿/电平/超时) | 按键/定时器/外部消息 |
| 时间约束 | 硬实时(必须及时响应) | 软实时(可以缓冲) |
| 典型状态数 | 5~10 | 3~7 |
| 多实例需求 | 常见(板上有 2 个 DHT11) | 少见 |
| 调试关注点 | 时序波形、超时计数 | 状态流转日志、用户行为 |

## 2. 版本一:逻辑构建(Switch-Case 直接映射)

### 2.1 适用场景

- 状态数量 ≤ 5
- 时序简单,单一通信流程
- 快速原型验证

### 2.2 原理讲解:DHT11 时序拆解

以 DHT11 温湿度传感器为例,其通信时序:

```
主机拉低18ms → 主机释放 → DHT11拉低80us → DHT11拉高80us → 40bit数据 → 结束
    ↓              ↓            ↓              ↓            ↓
 STATE_START   STATE_WAIT   STATE_RESP_L   STATE_RESP_H  STATE_DATA
```

**形象比喻**:像**接力赛**,每个选手(状态)只负责跑自己的一棒(时序段),跑完交接棒(状态切换)给下一个。

### 2.3 操作步骤清单

- [ ] Step 1:画时序图——标注每段持续时间(us/ms)和结束条件(电平跳变/超时)
- [ ] Step 2:定义状态枚举——每段时序 = 一个状态,必须包含 IDLE 和 COMPLETE
- [ ] Step 3:定义返回值枚举——BUSY/OK/ERROR_xxx(超时、校验失败等)
- [ ] Step 4:实现状态机主函数——switch(current_state) 结构,每个 case 检查退出条件
- [ ] Step 5:实现中断处理(可选)——捕获边沿跳变,中断只更新状态变量
- [ ] Step 6:添加超时保护——每个非 IDLE 状态都需要超时检测,超时回 IDLE

### 2.4 代码模板

```c
/* Step 2: 定义状态 */
typedef enum {
    DRV_STATE_IDLE,        /* 空闲 */
    DRV_STATE_START,       /* 起始信号 */
    DRV_STATE_WAIT_ACK,    /* 等待响应 */
    DRV_STATE_DATA,        /* 数据传输 */
    DRV_STATE_COMPLETE     /* 完成 */
} drv_state_t;

/* Step 3: 定义返回值 */
typedef enum {
    DRV_STATUS_IDLE,       /* 空闲 */
    DRV_STATUS_BUSY,       /* 忙 */
    DRV_STATUS_OK,         /* 成功 */
    DRV_STATUS_TIMEOUT,    /* 超时 */
    DRV_STATUS_ERROR       /* 错误 */
} drv_status_t;

/* Step 4: 状态机主函数 */
static drv_state_t g_state = DRV_STATE_IDLE;
static uint32_t g_timeout_cnt = 0;

drv_status_t Drv_Task(void)
{
    switch (g_state)
    {
    case DRV_STATE_IDLE:
        return DRV_STATUS_IDLE;

    case DRV_STATE_START:
        if (/* 起始信号完成条件 */)
        {
            g_timeout_cnt = 0;
            g_state = DRV_STATE_WAIT_ACK;
        }
        /* Step 6: 超时检测 */
        if (++g_timeout_cnt > TIMEOUT_START)
        {
            g_state = DRV_STATE_IDLE;
            return DRV_STATUS_TIMEOUT;
        }
        return DRV_STATUS_BUSY;

    case DRV_STATE_WAIT_ACK:
        if (/* 收到响应 */)
        {
            g_state = DRV_STATE_DATA;
        }
        if (++g_timeout_cnt > TIMEOUT_ACK)
        {
            g_state = DRV_STATE_IDLE;
            return DRV_STATUS_TIMEOUT;
        }
        return DRV_STATUS_BUSY;

    case DRV_STATE_DATA:
        if (/* 数据传输完成 */)
        {
            g_state = DRV_STATE_COMPLETE;
        }
        return DRV_STATUS_BUSY;

    case DRV_STATE_COMPLETE:
        g_state = DRV_STATE_IDLE;
        return DRV_STATUS_OK;

    default:
        g_state = DRV_STATE_IDLE;
        return DRV_STATUS_ERROR;
    }
}
```

### 2.5 V1 版的局限性

- **全局变量**:g_state 和 g_data 暴露在外,任何函数都能篡改
- **单实例**:只能驱动一个传感器,无法复用
- **状态切换无生命周期**:退出 WAIT_ACK 时忘了关中断?没人帮你兜底
- **无通知机制**:状态机完成了,应用层怎么知道?只能轮询返回值

> [!TIP]
> 这些痛点正是 [[有限状态机演进详解]] 中 V4(Enter/Exit)、V5(结构体封装)、V7(消息队列)要解决的问题。

## 3. 版本二:功能完善(结构体封装 + ENTRY/EXIT)

### 3.1 适用场景

- 多实例驱动(板上有 2 个 DHT11)
- 状态切换时需要资源管理(开关中断、使能外设)
- 需要可移植性

### 3.2 原理讲解:结构体封装

**形象比喻**:V1 的代码像**独生子**,全局变量与代码绑定。V2 是**印刷模版**,一个模版可以"印"出多个独立运行的实例。

```c
/* V1: 全局变量,只能一个实例 */
static uint8_t g_state;
static uint8_t g_data[5];

/* V2: 结构体封装,支持多实例 */
typedef struct {
    uint8_t state;
    uint8_t data[5];
    GPIO_TypeDef *port;
    uint16_t pin;
} drv_inst_t;

drv_inst_t sensor1, sensor2;   /* 两个独立实例 */
```

这与 [[有限状态机演进详解]] 中 **V5 的面向对象封装**是同一思想:self 指针让同一套代码操作不同实例的数据。

### 3.3 原理讲解:ENTRY/EXIT 机制

进入新状态前执行 ENTRY(初始化),离开旧状态前执行 EXIT(清理):

```
┌─────────────────┐
│  STATE_A EXIT   │ ← 离开 A 时执行
└────────┬────────┘
         ▼
┌─────────────────┐
│  STATE_B ENTRY  │ ← 进入 B 时执行
└────────┬────────┘
         ▼
┌─────────────────┐
│  STATE_B 主逻辑  │
└─────────────────┘
```

**驱动层 ENTRY/EXIT 典型用途**(对比 [[应用层状态机实操指南]]):

| 状态切换 | ENTRY 动作 | EXIT 动作 |
|---|---|---|
| 进入起始信号 | GPIO 配为输出,拉低总线 | — |
| 进入等待响应 | GPIO 配为输入,使能外部中断 | 关闭外部中断 |
| 进入数据接收 | 启动定时器计时脉冲宽度 | 停止定时器 |
| 进入空闲 | — | 关闭所有外设、释放引脚 |

### 3.4 操作步骤清单

- [ ] Step 1:定义驱动实例结构体(状态变量、数据缓冲区、硬件引脚配置、超时计数器)
- [ ] Step 2:修改所有函数签名——第一个参数改为 `drv_inst_t *self`,全局变量改 `self->xxx`
- [ ] Step 3:实现状态切换函数 `Drv_Tran(self, new_state)`,内含 EXIT 和 ENTRY 调用
- [ ] Step 4:定义 ENTRY/EXIT 处理(函数指针数组或 switch-case)
- [ ] Step 5:初始化函数 `Drv_Init(self, port, pin, ...)`,配置硬件,状态置 IDLE

### 3.5 代码模板

```c
/* Step 1: 驱动实例结构体 */
typedef struct drv_inst {
    drv_state_t state;
    uint8_t data[5];
    uint32_t timeout_cnt;
    /* 硬件配置 */
    GPIO_TypeDef *port;
    uint16_t pin;
    TIM_TypeDef *tim;
} drv_inst_t;

/* Step 3: 状态切换函数(含 ENTRY/EXIT) */
static void Drv_Tran(drv_inst_t *self, drv_state_t new_state)
{
    /* EXIT: 离开旧状态的清理工作 */
    switch (self->state)
    {
    case DRV_STATE_DATA:
        /* 关闭外部中断 */
        EXTI_Cmd(self->exti_line, DISABLE);
        break;
    default:
        break;
    }

    /* 更新状态 */
    self->state = new_state;
    self->timeout_cnt = 0;

    /* ENTRY: 进入新状态的初始化工作 */
    switch (new_state)
    {
    case DRV_STATE_START:
        /* GPIO 输出,拉低总线 */
        GPIO_SetOutput(self->port, self->pin);
        GPIO_Write(self->port, self->pin, 0);
        break;
    case DRV_STATE_WAIT_ACK:
        /* GPIO 输入,使能外部中断 */
        GPIO_SetInput(self->port, self->pin);
        EXTI_Cmd(self->exti_line, ENABLE);
        break;
    default:
        break;
    }
}

/* Step 2: 状态机主函数(带 self 指针) */
drv_status_t Drv_Task(drv_inst_t *self)
{
    switch (self->state)
    {
    case DRV_STATE_IDLE:
        return DRV_STATUS_IDLE;

    case DRV_STATE_START:
        if (/* 条件满足 */)
        {
            Drv_Tran(self, DRV_STATE_WAIT_ACK);
        }
        if (++self->timeout_cnt > TIMEOUT)
        {
            Drv_Tran(self, DRV_STATE_IDLE);
            return DRV_STATUS_TIMEOUT;
        }
        return DRV_STATUS_BUSY;

    /* ... 其他状态类似 ... */
    }
}

/* Step 5: 初始化函数 */
void Drv_Init(drv_inst_t *self, GPIO_TypeDef *port, uint16_t pin)
{
    self->port = port;
    self->pin = pin;
    self->state = DRV_STATE_IDLE;
    self->timeout_cnt = 0;
    memset(self->data, 0, sizeof(self->data));
}
```

### 3.6 V2 的改进与遗留问题

**改进**:
- 多实例:sensor1 和 sensor2 互不干扰
- 资源安全:EXIT 自动关中断/停定时器,不会"忘了关"
- 可移植:硬件配置通过参数传入,不硬编码

**遗留**:
- 应用层如何知道驱动完成了?仍需轮询返回值
- 中断里直接改状态变量,与主循环存在竞争风险
- 缺少统一的事件通知机制

## 4. 版本三:系统集成(中断解耦 + 异步通知)

### 4.1 适用场景

- 与 RTOS 集成
- 需要从中断中安全通知应用层
- 多驱动协同工作

### 4.2 原理讲解:中断与主循环解耦

**问题**:中断中不能执行耗时操作,但驱动状态机可能需要复杂处理。

**解决方案**:中断只做两件事:
1. 捕获硬件事件(记录时间戳、电平状态)
2. 设置标志位或更新状态变量

主循环负责状态机逻辑处理。

```
┌─────────────┐     标志位/队列     ┌─────────────┐
│   中断ISR   │ ─────────────────→ │  主循环Task  │
│  (10us内)   │                    │  (处理逻辑)  │
└─────────────┘                    └─────────────┘
```

**形象比喻**:中断像**前台接待**,只负责登记来客信息;主循环像**后台经理**,根据登记信息处理业务。

这与 [[有限状态机基础入门]] 中 4.3 节的"消息队列异步缓冲"以及 [[有限状态机演进详解]] 中 V7 的设计思想完全一致。

### 4.3 原理讲解:回调通知机制

驱动层不应直接调用应用层函数(打破分层),而是通过**回调函数**通知上层:

```c
/* 驱动层定义回调类型 */
typedef void (*drv_callback_t)(drv_status_t status, void *data);

/* 应用层注册回调 */
Drv_SetCallback(&sensor, on_sensor_complete);

/* 驱动层在状态机完成时调用 */
if (self->callback)
{
    self->callback(DRV_STATUS_OK, self->data);
}
```

**回调 vs 轮询 vs 队列**(参考 [[有限状态机基础入门]] 第 4 节):

| 方式 | 实时性 | 安全性 | 适用场景 |
|---|---|---|---|
| 轮询返回值 | 低(延迟取决于轮询间隔) | 高 | 简单场景 |
| 回调函数 | 高(完成即通知) | 中(注意回调上下文) | 裸机多模块 |
| 消息队列 | 中(入队→出队) | 高(异步缓冲) | RTOS、复杂系统 |

> [!CAUTION]
> **裸机回调上下文风险**:如果回调在中断中被调用,回调函数里不能做 printf、delay、malloc 等操作。安全的做法是回调在**主循环上下文**中执行,而非中断中。

### 4.4 操作步骤清单

- [ ] Step 1:分离中断处理与状态机逻辑——中断只记录时间戳/电平,设置 `self->irq_pending = 1`
- [ ] Step 2:状态机检查中断标志——`if (self->irq_pending) { 处理; self->irq_pending = 0; }`
- [ ] Step 3:定义回调函数类型——结构体增加 callback 字段
- [ ] Step 4:提供回调注册接口——`Drv_SetCallback(self, callback)`
- [ ] Step 5:状态机完成时触发回调——在 COMPLETE 或错误状态调用,回调在主循环上下文执行
- [ ] Step 6:RTOS 环境下用消息队列替代标志位——`xQueueSendFromISR()` / `xQueueReceive()`

### 4.5 代码模板

```c
/* Step 3: 回调类型定义 */
typedef void (*drv_callback_t)(drv_status_t status, uint8_t *data, uint8_t len);

/* 增强的驱动实例结构体 */
typedef struct drv_inst {
    drv_state_t state;
    uint8_t data[5];
    uint32_t timeout_cnt;

    /* 中断相关(必须 volatile) */
    volatile uint8_t irq_pending;       /* 中断标志 */
    volatile uint32_t irq_timestamp;    /* 中断时间戳 */
    volatile uint8_t irq_level;         /* 中断时电平 */

    /* 回调相关 */
    drv_callback_t callback;
    void *user_data;

    /* 硬件配置 */
    GPIO_TypeDef *port;
    uint16_t pin;
} drv_inst_t;

/* Step 4: 回调注册接口 */
void Drv_SetCallback(drv_inst_t *self, drv_callback_t cb, void *user_data)
{
    self->callback = cb;
    self->user_data = user_data;
}

/* Step 1: 中断处理(极简) */
void Drv_IrqHandler(drv_inst_t *self)
{
    /* 只记录关键信息,不做逻辑判断 */
    self->irq_timestamp = Timer_GetUs();
    self->irq_level = GPIO_Read(self->port, self->pin);
    self->irq_pending = 1;    /* 通知主循环 */
}

/* Step 2 & 5: 状态机主函数 */
drv_status_t Drv_Task(drv_inst_t *self)
{
    drv_status_t result;

    switch (self->state)
    {
    case DRV_STATE_DATA:
        /* Step 2: 检查中断标志 */
        if (self->irq_pending)
        {
            self->irq_pending = 0;
            uint32_t pulse_width = self->irq_timestamp - self->last_timestamp;
            /* ... 数据解析逻辑 ... */

            if (/* 数据接收完成 */)
            {
                Drv_Tran(self, DRV_STATE_COMPLETE);
            }
        }
        /* 超时检测 */
        if (++self->timeout_cnt > TIMEOUT)
        {
            Drv_Tran(self, DRV_STATE_IDLE);
            result = DRV_STATUS_TIMEOUT;
            goto notify;
        }
        return DRV_STATUS_BUSY;

    case DRV_STATE_COMPLETE:
        Drv_Tran(self, DRV_STATE_IDLE);
        result = DRV_STATUS_OK;
        goto notify;

    /* ... 其他状态 ... */
    }
    return DRV_STATUS_BUSY;

notify:
    /* Step 5: 触发回调通知应用层(主循环上下文,可安全操作) */
    if (self->callback)
    {
        self->callback(result, self->data, sizeof(self->data));
    }
    return result;
}

/* 应用层使用示例 */
void on_sensor_complete(drv_status_t status, uint8_t *data, uint8_t len)
{
    if (status == DRV_STATUS_OK)
    {
        printf("Temp: %d.%d\r\n", data[2], data[3]);
    }
    else
    {
        printf("Sensor error: %d\r\n", status);
    }
}

int main(void)
{
    drv_inst_t sensor;
    Drv_Init(&sensor, GPIOC, GPIO_Pin_3);
    Drv_SetCallback(&sensor, on_sensor_complete, NULL);

    Drv_Start(&sensor);

    while (1)
    {
        Drv_Task(&sensor);   /* 非阻塞轮询 */
        /* ... 其他任务 ... */
    }
}
```

### 4.6 RTOS 集成简化版

当使用 FreeRTOS 时,可用队列替代 volatile 标志位,获得阻塞等待和线程安全:

```c
/* FreeRTOS 版本 */
QueueHandle_t g_drv_queue;

void App_SensorTask(void *param)
{
    drv_msg_t msg;
    while (1)
    {
        /* 阻塞等待,不浪费 CPU */
        if (xQueueReceive(g_drv_queue, &msg, portMAX_DELAY))
        {
            App_ProcessSensorData(&msg);
        }
    }
}

void Drv_IrqHandler(drv_inst_t *self)
{
    drv_msg_t msg;
    msg.status = DRV_STATUS_OK;
    memcpy(msg.data, self->data, 5);

    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xQueueSendFromISR(g_drv_queue, &msg, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
```

## 5. 超时保护:驱动层的生命线

### 5.1 为什么每个状态都要超时?

硬件不可靠:传感器可能掉线、总线可能短路、中断可能丢失。如果状态机卡在某个"等待"状态,整个系统就死了。

**铁律**:除 IDLE 外,每个状态都必须有超时退出路径。

### 5.2 超时计数器的实现方式

```c
/* 方式 1:主循环累加(依赖心跳周期已知) */
if (++self->timeout_cnt > TIMEOUT_MS)
{
    Drv_Tran(self, DRV_STATE_IDLE);
    return DRV_STATUS_TIMEOUT;
}

/* 方式 2:记录进入时间戳(更精确,不受心跳频率影响) */
uint32_t elapsed = Timer_GetMs() - self->entry_timestamp;
if (elapsed > TIMEOUT_MS)
{
    Drv_Tran(self, DRV_STATE_IDLE);
    return DRV_STATUS_TIMEOUT;
}
```

方式 2 更精确,但需要硬件定时器支持。方式 1 更简单,在 [[有限状态机基础入门]] 第 5 节的"心跳驱动"模型中自然适配。

### 5.3 超时值的选取原则

| 依据 | 超时倍数 | 示例(DHT11) |
|---|---|---|
| 协议规定最大时长 | 2~3 倍 | 响应 80us → 超时 200us |
| 无明确规范 | 5~10 倍典型值 | 起始 18ms → 超时 50ms |
| 安全关键 | 硬性限制 | 总通信超时 100ms(无论哪步) |

## 6. 中断驱动型 vs 轮询驱动型

驱动层状态机有两种典型的驱动方式:

### 6.1 中断驱动型

**特征**:状态切换由硬件中断触发,主循环只做数据处理。

**适用**:I2C/SPI(起始/停止条件由中断捕获)、UART DMA(传输完成中断)。

```c
/* 中断中推进状态 */
void EXTI_Callback(void)
{
    switch (g_state)
    {
    case DRV_STATE_WAIT_ACK:
        Drv_Tran(&sensor, DRV_STATE_DATA);
        break;
    case DRV_STATE_DATA:
        /* 解析脉冲宽度 */
        break;
    }
}
```

### 6.2 轮询驱动型

**特征**:主循环定时调用 `Drv_Task()`,在函数内检查硬件状态并推进状态机。

**适用**:DHT11(需要精确延时控制起始信号)、1-Wire(时序由软件控制)。

```c
/* 主循环中轮询推进 */
while (1)
{
    Drv_Task(&sensor);   /* 每次调用推进一个 tick */
}
```

### 6.3 混合型(推荐)

中断捕获边沿事件,主循环处理逻辑——兼顾实时性和安全性:

```c
/* 中断:只记录事件 */
void EXTI_Callback(void)
{
    sensor.irq_timestamp = DWT->CYCCNT;
    sensor.irq_pending = 1;
}

/* 主循环:处理事件 + 推进状态 */
Drv_Task(&sensor);
```

## 7. 常见陷阱与最佳实践

> [!WARNING]
> **陷阱 1:volatile 遗漏**
> 中断与主循环共享的变量(irq_pending、irq_timestamp 等)必须加 volatile,否则编译器优化可能导致主循环读不到中断写入的值。

> [!WARNING]
> **陷阱 2:回调在中断上下文执行**
> 如果 Drv_Task() 在中断中被调用,则回调也在中断上下文。回调里绝对不能做 printf、delay、malloc。解决方案:回调只设置标志位,主循环中检查标志位并执行真正的处理。

> [!WARNING]
> **陷阱 3:多实例共用中断号**
> STM32 的 EXTI 线与 GPIO 引脚一一绑定(如 PA0/PB0/PC0 共用 EXTI0)。如果两个传感器分别接 PA0 和 PB0,它们会触发同一个中断。中断处理函数中必须区分是哪个实例。

> [!CAUTION]
> **陷阱 4:超时计数器溢出**
> 如果 timeout_cnt 是 uint16_t 且心跳周期 1ms,最大计时 65.5 秒。需要更长时间的应用应使用 uint32_t 或时间戳方式。

> [!IMPORTANT]
> **最佳实践 1:先画时序图再写代码**
> 驱动层状态机的状态划分完全来自时序图。不画图直接写码,极易遗漏状态或超时。

> [!IMPORTANT]
> **最佳实践 2:每个状态一个超时**
> 除 IDLE 外,所有状态都必须有超时退出路径。这是驱动层区别于应用层的关键要求。

> [!IMPORTANT]
> **最佳实践 3:中断只记录,主循环只处理**
> 严格分离中断与逻辑,保证中断极速返回,避免优先级反转。

## 8. 版本选型指南

| 版本 | 状态数 | 实例数 | RTOS | 推荐场景 |
|---|---|---|---|---|
| V1 逻辑构建 | ≤5 | 1 | 否 | 快速原型、简单协议 |
| V2 功能完善 | 5~10 | 多个 | 否 | 产品级驱动、多传感器 |
| V3 系统集成 | 不限 | 多个 | 是 | 复杂系统、RTOS 集成 |

**选型建议**:
- 简单传感器(DHT11、DS18B20)→ V1 即可
- 多路同类传感器 → V2(struct 封装 + Entry/Exit)
- RTOS 环境 + 多驱动协同 → V3(回调/队列通知)

## 9. 总结速查

- **核心映射**:时序图的每段等待 = 一个状态,电平跳变 = 事件
- **三版递进**:V1(switch-case 直接映射)→ V2(封装 + 生命周期)→ V3(异步通知)
- **超时铁律**:除 IDLE 外每个状态必须有超时退出
- **中断铁律**:中断只记录,主循环只处理
- **通知方式**:轮询返回值 → 回调函数 → 消息队列,按复杂度递增
- **多实例关键**:self 指针 + 结构体封装,代码共享(Flash)、数据隔离(RAM)
