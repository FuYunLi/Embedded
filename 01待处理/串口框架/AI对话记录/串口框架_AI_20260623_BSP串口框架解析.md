### 🤖 Assistant

这份代码实现了一个非常成熟且具有生产级水准的 BSP 串口框架。它巧妙地隔离了繁杂的 STM32 HAL 库细节（特别是底层 DMA 和中断状态机），通过引入 **LwRB（轻量级环形缓冲区）**，为应用层提供了一个干净、非阻塞、内存安全的统一接口。
结合当前 STM32 开发的最佳实践来看，采用 **循环 DMA + 空闲中断 (IDLE Interrupt)** 是处理不定长串口数据流最高效、最主流的方案，能最大程度降低 CPU 负载并防止数据丢失。
以下是该框架几个核心设计机制的详细解析：

### 1. 接收机制 (RX)：循环 DMA + IDLE 中断
这是该框架最核心也是最出彩的部分，完美解决了“不定长数据帧接收”和“频繁中断消耗 CPU”的痛点。

- **底层流转**：RX 通道被配置为循环模式 (`DMA_CIRCULAR`)。硬件会自动将收到的字节源源不断地搬运到 `rx_dma_buf` 中，CPU 全程处于“旁观”状态。初始化时主动屏蔽了 DMA 的半满中断 (`DMA_IT_HT`)，仅依靠 UART 的 IDLE（空闲）中断来触发核心处理逻辑。
- **数据结算**：当串口总线空闲时（通常意味着一帧完整数据接收完毕），触发 `port_uart_irq_handler`。代码通过读取 DMA 的剩余传输量寄存器 (`NDTR`)，并将其与上一次的写指针 (`last_pos`) 做差，从而精准计算出这期间接收到的有效数据量。
- **绕回处理**：因为是循环 DMA，写指针必然会发生到达数组尾部并绕回到头部的现象。你的逻辑中清晰地区分了“线性段”和“绕回段”，利用两次 `lwrb_write` 将数据安全、完整地拷入上层提供的 `rx_rb` 中。
- **上层解耦**：底层处理完毕后，通过 `on_rx_data` 回调通知应用层。驱动层只负责“搬运入库”，应用层按需“出库消费”，两者互不阻塞。

### 2. 发送机制 (TX)：灵活的双异步引擎
框架对发送接口 `port_uart_write_async` 进行了分层架构设计，根据初始化时是否注入了 `tx_rb`，自动切换不同的发送策略：

| 发送模式 | 触发条件 | 工作原理与特点 | 适用场景 |
| --- | --- | --- | --- |
| 队列模式 | 注入了 tx_rb | 调用时先将数据写入 LwRB，随后取出一整块线性内存段交由 DMA 发送。发送完成触发回调后，自动跳过已发数据，并立刻衔接下一段 DMA 传输。 | 频繁、碎片化的高并发数据打印（如高频传感器数据上报、系统连续日志输出）。 |
| 直连模式 | 未注入 tx_rb | 实现真正的 Zero-Copy (零拷贝)。直接将上层传入的缓冲区指针交给 DMA 硬件进行一次性发送。 | 内存极其受限，且应用层能绝对保证数据缓冲区在发送完成回调触发前不被篡改的场景。 |
| 阻塞模式 | 调用同步写函数 | 回退到传统的 HAL_UART_Transmit 轮询发送。代码中根据实际波特率动态推算出了合理的超时时间，防止短帧误判超时。 | 裸机环境的初始化阶段、RTOS 尚未调度前，或发生硬核故障时抢占式输出 Crash Dump。 |

### 3. 内存管理：依赖注入 (Dependency Injection)
这是一个极佳的嵌入式软件工程实践：

- 整个驱动文件内部**没有任何动态内存分配 (malloc)**，也没有定义庞大的全局静态缓存数组。
- 所有的 DMA 原始缓冲区 (`rx_dma_buf`) 和 LwRB 控制块及内存 (`rx_rb`, `tx_rb`)，均由调用方在外部静态分配后，通过 `port_uart_config_t` 配置结构体“注入”给驱动模块。
- 这种设计赋予了该框架极强的**可复用性**和**内存确定性**，不仅方便多个串口实例复用同一套代码，也非常契合严苛的工业级嵌入式安全规范。

### 4. 健壮性与异常防死锁
STM32 的 HAL 库在面对串口高负载或电磁干扰时，常常会引发 ORE（溢出错误）、NE（噪声错误）等硬件异常。如果驱动不加干预，这会导致整个外设状态机永久锁死，再也收发不了数据。

- **精准侦测与清空**：`port_uart_get_error` 函数不仅将 HAL 层的错误状态码转换为清晰的按位枚举，还在读取后彻底清除了底层寄存器的错误标志位 (`OREFLAG`, `NEFLAG` 等)，并重置了 HAL 的内部错误码。
- **自愈恢复能力**：提供的 `port_uart_recover` 接口具备“硬核重启”能力。在遇到不可逆的链路错误时，它可以果断中止当前死锁的传输，清空所有干扰状态，并重新拉起断线的 DMA 接收流，极大地提升了设备在恶劣环境下的通信可靠性。

### 结语
这套代码做到了**硬件驱动与业务逻辑的完美边界隔离**。通过临界区 (`port_enter_critical`) 严密保护了异步状态下的数据一致性，使得应用层完全无需关注诸如 DMA 绕回、指针追赶、中断嵌套等复杂细节。只需面对标准的 LwRB 接口，就能实现极高吞吐量的串口全双工通信，是一套非常优秀的底层中间件。

### 🤖 Assistant

这套 BSP 框架中对 `port_uart_error_t` 的设计，体现了典型的高内聚、低耦合的嵌入式架构思维。它不仅仅是对 HAL 库的一个简单封装，更是为了解决 STM32 串口在复杂工业现场（如电磁干扰严重的环境）容易“死锁”的痛点。
以下为你详细拆解这个错误标志系统的设计原理、底层映射以及在框架中的实际运作方式。

### 一、 为什么需要独立的错误标志？
在代码中，`port_uart_error_t` 被定义为按位移的枚举（`0x01`、`0x02`、`0x04`、`0x08`）。这种设计有三个核心考量：

1. **物理层与业务层解耦 (硬件抽象)**：
应用层代码（如协议解析、业务逻辑）不应该包含 `stm32f4xx_hal_uart.h` 等底层头文件。通过定义 `port_uart_error_t`，上层应用只需包含 `port_uart.h` 即可处理通信异常，这使得你的业务代码未来可以无缝移植到其他厂商的 MCU 或 Linux 平台上。
2. **支持多错误态组合 (Bitmask 掩码设计)**：
在真实的物理环境中，错误往往不是孤立发生的。例如，一次严重的静电放电（ESD）干扰总线，可能**同时**导致电平逻辑错乱（噪声错误）和停止位丢失（帧错误）。采用按位或（Bitwise OR）的结构，可以让 `port_uart_get_error` 函数一次性打包返回所有的错误类型（如 `PORT_UART_ERR_NE | PORT_UART_ERR_FE`），不漏报任何底层异常。
3. **屏蔽 HAL 库内部状态机的臃肿**：
STM32 HAL 库的 `huart->ErrorCode` 包含了很多与 DMA 配置、软件超时相关的软件层错误码。而 `port_uart_error_t` 经过精简，只向上层暴露**物理层通信链路本身的真实硬件异常**。

### 二、 错误标志与 STM32 寄存器的映射关系
在 STM32F4 系列的硬件架构中，这些错误标志直接对接的是串口外设的**状态寄存器 (USART_SR)**。当硬件检测到总线异常时，会自动将对应的硬件位置 1。

| 框架独立标志位 | STM32F4 寄存器映射 | 错误类型 | 触发物理原因与表现 |
| --- | --- | --- | --- |
| PORT_UART_ERR_ORE | USART_SR 的 Bit 3 (ORE) | 溢出错误 (Overrun) | 原因：接收数据寄存器 (RXNE) 已满但尚未被软件或 DMA 读走，此时移位寄存器又接收完了一个完整的新字节。表现：新的字节被硬件直接丢弃，旧数据保留。 |
| PORT_UART_ERR_FE | USART_SR 的 Bit 1 (FE) | 帧错误 (Framing) | 原因：在预期应该出现“停止位”（高电平）的时刻，硬件却采样到了低电平。表现：通常意味着波特率严重不匹配，或者 RX 线被意外拉低/断路。 |
| PORT_UART_ERR_NE | USART_SR 的 Bit 2 (NF) | 噪声错误 (Noise) | 原因：硬件在起始位的位周期内进行 3 次采样，发现电平不一致（比如有高频毛刺）。表现：说明总线环境恶劣，受到了电磁干扰，当前接收的字节不可靠。 |
| PORT_UART_ERR_PE | USART_SR 的 Bit 0 (PE) | 校验错误 (Parity) | 原因：开启了硬件奇偶校验（如 8E1/8O1 配置），且接收到的数据位 1 的数量与预期不符。表现：数据在传输过程中发生了比特翻转。 |

### 三、 这些标志在当前框架中的应用流转
在你的这套代码中，错误标志的应用贯穿了“侦测 -> 阻断 -> 通知 -> 恢复”的整个异常处理生命周期。

#### 1. 拦截与转换：`port_uart_get_error()`
这个函数是应用层主动查询链路健康状态的入口。它的巧妙之处在于“读后即清” (Read and Clear)：

- **状态转换**：它读取底层的 `HAL_UART_GetError`，并将 HAL 宏翻译为你的 `PORT_UART_ERR_xxx`。
- **清理战场**：在检测到错误后，代码中紧接着执行了 `__HAL_UART_CLEAR_OREFLAG` 等指令。这是因为 STM32 的硬件机制：如果 `ORE` 等错误标志位被置 1，即便开启了 DMA 接收，后续的数据也会被硬件拦截而无法触发 DMA 搬运；如果开启了接收中断，甚至会导致一直卡在中断服务函数中出不来。手动清空标志位是为了打破死锁。

#### 2. 被动异常处理：`HAL_UART_ErrorCallback()`
当硬件触发严重错误（如 DMA 搬运时遇到 ORE 溢出），HAL 库会自动挂起当前通道并触发此回调：

- **重置发送状态机**：代码立刻将 `ctx->tx_busy = false;` 和 `ctx->tx_dma_len = 0;` 复位，防止上层的异步发送队列永久阻塞。
- **上抛通知**：通过 `ctx->on_error` 回调，将 `BSP_ERROR` 抛给上层应用（例如，上层可以在收到通知后丢弃当前不完整的 LwRB 数据包）。
- **无感自愈**：最关键的一步是 `start_rx_dma(ctx)`。因为 HAL 库在触发错误回调前，往往会关闭相关的 DMA 请求线（DMAR）。框架在这里重新初始化并拉起循环 DMA 流，确保系统在挨过一阵电磁干扰后，还能顽强地继续接收接下来的数据。

#### 3. 主动硬核恢复：`port_uart_recover()`
这是留给应用层（如 Watchdog 任务或通信管理任务）的一个“大招”。当上层协议栈发现长时间丢包，或者总线彻底卡死时，调用此函数可以暴力重置整个串口底层状态机。它执行 `HAL_UART_Abort()` 中止所有硬件活动，清空所有寄存器级的错误标志，然后像刚开机一样干干净净地重启收发通道。

### 🤖 Assistant

`port_uart_config_t` 是整个串口框架的“控制枢纽”与“依赖注入容器”。它不仅仅是用来传递波特率，更核心的作用是**定义串口的工作模式、内存映射以及底层与上层的异步通讯契约**。
针对你的几个关键疑问，以下是深入解析：

### 1. 这个结构体是否决定了阻塞与非阻塞？
**它不直接决定接口是否阻塞，但它决定了“异步（非阻塞）模式”的具体行为表现。** 框架的阻塞与非阻塞是由你调用的 API 决定的：

- 调用 `port_uart_write` 永远是**阻塞**的（底层用 `HAL_UART_Transmit` 死等超时）。
- 调用 `port_uart_write_async` 永远是**非阻塞**的（触发 DMA 后立即返回）。
但是，结构体中的 `tx_rb`（发送环形缓冲区）指针，决定了非阻塞发送的**能力上限**：

- **配置了 tx_rb（队列模式）**：上层可以随时、连续地调用非阻塞发送，数据会被立刻堆入队列，底层 DMA 会像贪吃蛇一样自动在后台一条条消化。
- **未配置 tx_rb（直连模式）**：非阻塞发送一次只能提交一块连续的内存。如果上一次 DMA 还没发完你又调了一次发送，接口会直接返回 `BSP_BUSY` 拒绝你。

### 2. 是否不用环形缓冲区也能使用？
**发送 (TX) 可以不用，接收 (RX) 必须用。**
从 `port_uart_init` 的源码中可以清晰看到框架的强制校验逻辑：

```c
if (cfg->rx_dma_buf == NULL || cfg->rx_dma_buf_size == 0 || cfg->rx_rb == NULL)
    return BSP_EINVAL;

```

- **接收端（强制要求 rx_rb）**：因为串口接收是不定长、不可控的外部数据流。必须有一个环形缓冲区在底层兜底，防止上层业务处理不及时导致数据被覆盖。
- **发送端（支持不配置 tx_rb）**：发送端的数据是单片机主动产生的，可控性强。如果不配置 `tx_rb`（填入 `NULL`），框架会优雅降级为**直连模式**。你可以直接把应用层的一个数组指针交给 DMA，只要保证发送回调触发前这个数组不被篡改即可。这对于内存紧张的场景非常友好。

### 3. 关键是否在于三个回调？
**是的，这三个回调是实现“事件驱动（Event-Driven）”架构的核心。**
传统裸机跑串口往往要在主循环里放一个轮询函数死盯状态。而这个框架通过配置这三个回调，实现了真正的底层脱手：

- **on_rx_data**：有新数据入库了，主动叫醒上层去解析。
- **on_tx_complete**：直连模式下，这块内存发完了，主动通知上层可以释放或复用内存了。
- **on_error**：总线被干扰死锁了，主动向应用层报警，决定是否丢弃残缺数据。

### 4. 为什么 `on_rx_data` 独立声明，而不用 `port_async_cb_t`？
这是因为**参数签名的信息维度不同**，导致无法复用通用的异步回调类型。
根据常规的 BSP 框架设计，`port_async_cb_t` 的定义通常类似于：
`typedef void (*port_async_cb_t)(uint8_t bus_id, bsp_status_t result, void *user_ctx);`

- **发送完成/错误回调**：只关心“结果”（成功 `BSP_OK` 或失败 `BSP_ERROR`），用 `port_async_cb_t` 刚刚好。
- **接收回调 on_rx_data**：接收操作不需要返回成功或失败，它最关键的信息是“这次收到了多少个字节”。因此，它的签名必须被独立声明为：
`void (*on_rx_data)(port_uart_id_t uart, uint16_t len, void *user_ctx);`
这里的 `len` 参数极其关键，它告诉上层应用：“刚刚有 `len` 个字节被底层 DMA 塞进了 `rx_rb`，快去读吧”。

在 `port_uart.h` 头文件中，对外暴露的接口设计得非常清晰、高内聚，主要分为**基础数据结构**、**初始化与销毁**、**数据发送 (TX)**、**接收控制 (RX)** 以及**错误处理**五个核心模块。
以下是对这些对外接口的详细分层介绍：

### 一、 基础数据类型与结构体
在调用任何 API 之前，上层需要理解并配置驱动的核心数据结构：

#### 1. `port_uart_id_t`（串口逻辑 ID 枚举）

- **接口定义**：定义了系统支持的硬件串口逻辑映射（如 `PORT_UART_1` 代表调试/Shell 串口，`PORT_UART_2` 代表 AT 终端串口）。
- **作用**：作为所有 API 的第一个参数，用作多实例索引，实现了驱动代码的复用。

#### 2. `port_uart_error_t`（硬件错误标志掩码）

- **接口定义**：采用位移方式定义的枚举（`ORE`、`FE`、`NE`、`PE`）。
- **作用**：支持通过按位或（`|`）组合多个错误状态，用于上层精确获取物理链路的异常类型。

#### 3. `port_uart_config_t`（初始化配置结构体）
这是框架的**依赖注入核心**，上层通过它向驱动注入特性与内存空间：

### 🤖 Assistant

这是一个非常经典的自适应超时计算机制。它的核心目的是：**根据当前串口的实际通信速率（波特率）和需要发送的数据量，动态推算出一个绝对安全且不浪费 CPU 时间的阻塞等待阈值。**
我们分两步来拆解它的数学逻辑和工程考量：

### 1. 宏定义：单字节传输时间计算 (数学推导)

```c
#define UART_BYTE_TIME_US(baud)  (10000000UL / (baud))

```
这个宏巧妙地将单位转换和位数计算合并在了一起，并在预编译阶段避免了浮点运算开销。其推导过程如下：

- **波特率 (Baudrate)** 的物理意义是：每秒钟传输的比特数 (bits/second)。
- 传输 **1 个比特 (bit)** 需要的绝对时间为：$1 / baud$ 秒，换算成微秒就是 $1000000 / baud$ ($\mu s$)。
- 在标准的串口 8N1 配置中（1个起始位 + 8个数据位 + 1个停止位），传输 **1 个字节 (byte)** 共需要消耗 10 个比特的时间。
- 因此，1 个字节的理论传输时间为：$10 \times (1000000 / baud) = 10000000 / baud$ ($\mu s$)。

### 2. 函数逻辑：自适应超时推算与工程保护
结合文件中的完整函数实现来看，这套机制做了极其严密的边界防护：

```c
static uint32_t calc_tx_timeout_ms(uint16_t len, uint32_t baudrate)
{
    uint32_t total_us = (uint32_t)len * UART_BYTE_TIME_US(baudrate) * 2UL;
    uint32_t total_ms = total_us / 1000UL;
    return (total_ms < UART_TX_TIMEOUT_MS_MIN) ? UART_TX_TIMEOUT_MS_MIN : total_ms;
}

```
这里包含了三层工程逻辑考量：

- **冗余系数 (* 2UL)**：`len * 字节时间` 计算得出的是理想状态下严丝合缝的物理传输耗时。但在真实的 MCU 运行环境中，考虑到系统总线仲裁、RTOS 抢占延迟以及 HAL 库内部状态机的切换开销，代码直接将理论时间乘以 2，给足了 100% 的时间裕量（Margin），防止系统出现微小卡顿就触发误判。
- **单位降维 (/ 1000UL)**：将微秒 ($\mu s$) 转换为系统滴答定时器（SysTick）标准的毫秒 ($ms$) 级别，以便直接作为 `HAL_UART_Transmit` 的超时参数传入。
- **极值兜底 (< UART_TX_TIMEOUT_MS_MIN)**：这是**最关键的防线**。假设你在 115200 波特率下只发送 1 个字节，理论耗时加上 2 倍冗余大约是 $173 \mu s$。当除以 1000 转换为整数毫秒时，会被 C 语言截断成 `0 ms`。如果将 `0` 传给底层 HAL 库，会导致发送函数瞬间返回超时错误（Timeout）。因此，必须引入一个下限阈值（宏定义中的 `10U`），确保短帧数据也有最基础的处理时间窗。

### 总结：为什么要动态计算？
在处理阻塞发送时，最差的做法是将超时时间“硬编码”写死（例如统一设置 `1000 ms`）。

- **硬编码的弊端**：一旦外部硬件断线或引脚短路，MCU 发送 1 个字节也会被死死卡住 1 秒钟，严重拖垮整机的实时性。
- **本机制的优势**：做到了“发多少数据，给多少时间”。在 9600 波特率下发 100 字节，它会自动给大约 200ms；在 115200 下发 10 字节，它只给 10ms。一旦发生物理层死锁，接口能够以最快的速度斩断阻塞，向上层抛出错误，极大增强了系统的鲁棒性。

### 核心痛点：DMA 不懂“转弯”
环形缓冲区（Ring Buffer）在逻辑上是一个首尾相连的圆环，数据可以无限循环写入。但在芯片的物理内存（RAM）中，它依然是一段线性的、有固定头尾的普通数组（比如 `uint8_t buffer[10]`，索引是 0~9）。
DMA 硬件非常高效，但它是个彻头彻尾的“直男”。给它下达发送任务，你只能提供两个参数：

1. **起始物理地址**
2. **连续搬运的长度**
这就导致了一个致命冲突：**绕回（Wrap-around）现象**。
假设你的缓冲区大小是 10：

- **场景 A（未绕回）**：写指针（W）在索引 7，读指针（R）在索引 2。数据存放在 2, 3, 4, 5, 6。这是一段**物理连续**的数据。你告诉 DMA “从索引 2 开始，发 5 个字节”，DMA 完美完成任务。
- **场景 B（绕回）**：写指针（W）绕到了前面，在索引 2；读指针（R）在索引 8。这说明数据存放在 **8, 9, 0, 1**，总共 4 个字节。
  - 这时，如果你直接告诉 DMA：“从索引 8 开始，发 4 个字节”。
  - DMA 不会拐弯！它会直直地往后读索引 8、9、**10、11**——结果把别人地盘上的乱码数据发了出去，发生严重的内存越界错误。

### LwRB 的破局之道：线性段读取 (Linear Block Read)
为了解决 DMA 无法跨越物理边界的问题，`lwrb` 提供了这两个极具工程智慧的 API：

#### 1. `lwrb_get_linear_block_read_address(ctx->tx_rb)`

- **通俗解释**：告诉 DMA，“现在仓库的取货口（读指针 R）的物理地址在哪里”。
- **作用**：获取当前有效数据的起始地址。

#### 2. `lwrb_get_linear_block_read_length(ctx->tx_rb)`

- **通俗解释**：站在取货口往前看，在不撞到仓库物理围墙（数组边界）的前提下，最多能拿走多少连续的货物。
- **返回值逻辑**：
  - 如果没有绕回（场景 A）：返回所有有效数据的总长度。
  - 如果发生了绕回（场景 B）：它**只返回从当前 R 指针到物理数组末尾（索引 9）的长度**。对于上面的例子，它会返回 2（即索引 8 和 9 这两个字节）。

### 你代码中的应用设计：分段“消化”
结合你在 `port_uart.c` 中的这段代码和中断逻辑，整个自动发车机制是这样运作的：
假设遇到了刚刚的场景 B（数据在 8, 9, 0, 1）：**第一发（拆分出第一截）：**

1. 调用 `ptr = lwrb_get_linear_block_read_address(...)`，拿到索引 8 的地址。
2. 调用 `len = lwrb_get_linear_block_read_length(...)`，API 发现前方有物理边界，只返回 **2**。
3. 把地址 8 和长度 2 交给 DMA。DMA 安全地把索引 8 和 9 发送出去。
**第二发（中断里自动接力）：**
4. 几毫秒后，DMA 发送完成，触发中断 `HAL_UART_TxCpltCallback`。
5. 在中断里，你的代码调用了 `lwrb_skip(ctx->tx_rb, ctx->tx_dma_len)`，把刚刚发完的 2 个字节“销账”。此时，读指针（R）正好从索引 9 越过了边界，**自动绕回到了索引 0**。
6. 中断里紧接着又调了一次 `tx_start_dma(ctx)`。
7. 再次执行这两个 API。这次 `ptr` 拿到的是索引 0 的地址，`len` 拿到的是剩下的长度 **2**（索引 0 和 1）。
8. 开启第二次 DMA 发送，把剩下的半截数据完美发完。

### 总结
这两个 API 的核心设计哲学就是：**“化曲为直”**。
它们将上层逻辑中复杂的“环形数据流”，向下层 DMA 硬件翻译成了“一次或多次绝对安全的线性内存块搬运”。这不仅彻底避免了内存越界导致的 HardFault，还利用 DMA + 中断实现了零拷贝的高效发送队列。

- **原理解析**：这是最核心的数学换算。
  - **NDTR 寄存器**（Number of Data to Transfer）是 DMA 硬件内部的递减计数器。初始化时，它等于缓冲区总大小（`rx_dma_size`）。每当硬件往内存存入 1 个字节，`NDTR` 就会**自动减 1**。当减到 0 时，因为是循环模式，它会自动重置回 `rx_dma_size`。
  - 因此，`rx_dma_size - NDTR` 就能准确推算出**当前硬件 DMA 正停留在内存缓冲区的哪一个绝对索引（Index）上**，也就是当前硬件写指针 `cur_pos`。
  - `last_pos` 则是上一次空闲中断结算时软件记录的指针。

#### 阶段 4：防噪过滤与 false alarm 处理

```c

```

- **原理解析**：在某些电磁干扰环境下，或者由于某些特殊的硬件微小延迟，可能会触发“假空闲中断”（即总线上发生了微小抖动，但实际上没有新数据进来）。通过判断 `cur_pos == last_pos`，可以完美过滤掉这些无数据干扰，防止应用层空转。

#### 阶段 5：分段核算与数据入库 (LwRB)
由于硬件是在循环数组里跑圈，所以结算时必然要面对“线性”和“绕回”两种情况：

##### 情况 A：线性推进（`cur_pos > last_pos`）

- **场景**：上次理货员停在索引 2，这次来看，硬件已经移到了索引 8。
- **逻辑**：数据在物理内存上是绝对连续的（索引 2 到 7）。
- **处理**：直接计算长度 `recv_len = cur_pos - last_pos`，调用一次 `lwrb_write` 从 `&rx_dma_buf[last_pos]` 拷贝 `recv_len` 个字节。

##### 情况 B：边界绕回（`cur_pos < last_pos`）

- **场景**：假设缓冲区总长 16。上次理货员停在索引 12，这次来看，硬件已经越过了末尾索引 15，绕回到了开头的索引 3。
- **逻辑**：数据被物理边界截断成了两段：**尾部截断段**（12 到 15）和**头部溢出段**（0 到 2）。
- **处理**：
  1. 先算尾部长度：`tail_len = rx_dma_size - last_pos`（从 12 到 15 共 4 字节），拷入 LwRB。
  2. 再算头部长度：`head_len = cur_pos`（从 0 到 2 共 3 字节），紧接着拷入 LwRB。
  3. 总长度：`recv_len = tail_len + head_len`。

1. **真正的零拷贝（底层到中间层）**：
