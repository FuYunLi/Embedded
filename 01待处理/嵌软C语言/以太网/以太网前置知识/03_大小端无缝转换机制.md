# 大小端无缝转换机制

在以太网开发中，如果你发现单片机收到的 IP 地址或者端口号“看反了”，比如你发送到端口 `80` (0x0050)，但单片机解析出来却是 `20480` (0x5000)——这就是经典的**大小端冲突（Endianness Conflict）**问题。

---

## 1. 什么是大小端？

当我们定义一个 32 位（4 字节）的整数 `uint32_t val = 0x12345678;` 时，它在内存中的存储顺序有两种截然不同的方式：

* **大端序（Big-Endian）**：高位字节存放在低地址。符合人类的直观阅读习惯。
  * `[低地址] 0x12 -> 0x34 -> 0x56 -> 0x78 [高地址]`
* **小端序（Little-Endian）**：低位字节存放在低地址。符合计算机底层算术逻辑单元（ALU）的进位处理习惯。
  * `[低地址] 0x78 -> 0x56 -> 0x34 -> 0x12 [高地址]`

> [!CAUTION] 
> 绝大多数现代单片机（包括 ARM Cortex-M 和 CH32V307 RISC-V 内核）默认均采用**小端序（主机字节序，Host Byte Order）**。
> 然而，TCP/IP 协议明确规定，网络上传输的所有多字节数据首部（如 IP 地址、端口号等）必须使用**大端序（网络字节序，Network Byte Order）**！

---

## 2. 核心转换函数与宏

为了解决单片机（小端）与网络（大端）的矛盾，在发送网络包之前，以及解析接收到的网络包时，必须使用专门的转换宏。

POSIX 标准规定了四个经典的转换函数名（通常底层实现为高效的内联汇编或宏定义）：

| 函数名 | 含义 | 作用 |
|---|---|---|
| `htons()` | Host TO Network Short | 将 16 位短整型从主机字节序转为网络字节序（用于**端口号**转换） |
| `htonl()` | Host TO Network Long | 将 32 位长整型从主机字节序转为网络字节序（用于**IP地址**转换） |
| `ntohs()` | Network TO Host Short | 将 16 位短整型从网络字节序转为主机字节序 |
| `ntohl()` | Network TO Host Long | 将 32 位长整型从网络字节序转为主机字节序 |

> **规律速记**：以 `h` (host) 开头的一般用在**发送前构建包头**时；以 `n` (network) 开头的一般用在**接收后解析包头**时。

---

## 3. 手写大小端倒转的本质

如果我们不调用库函数，纯用 C 语言位运算如何实现一个 `htons`？其本质就是高低字节的**化整为零与重新拼接**：

```c
// 16位字节序翻转实现
uint16_t my_htons(uint16_t host_short) {
    return ((host_short & 0xFF00) >> 8) | 
           ((host_short & 0x00FF) << 8);
}

// 32位字节序翻转实现
uint32_t my_htonl(uint32_t host_long) {
    return ((host_long & 0xFF000000) >> 24) |
           ((host_long & 0x00FF0000) >> 8)  |
           ((host_long & 0x0000FF00) << 8)  |
           ((host_long & 0x000000FF) << 24);
}
```

> [!TIP] 
> 对于数据负载区域（Payload，比如你发送的字符串 "Hello" 或者自定义的结构体），除非协议特别声明，通常**不需要**进行字节序转换，按原样发送即可，仅协议的首部（Header）控制字段受大小端严格约束。
