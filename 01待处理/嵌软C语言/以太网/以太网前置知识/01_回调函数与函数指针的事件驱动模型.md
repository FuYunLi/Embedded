# 回调函数与函数指针的事件驱动模型

在以太网协议栈（如 **LwIP**）的无操作系统（NO_SYS=1）移植中，**事件驱动模型**是其核心架构。这种模型完全依赖于 C 语言的**函数指针**和**回调函数（Callback）**机制。

---

## 1. 为什么需要函数指针？

传统的单片机编程通常是“顺序执行”或“轮询（Polling）”模型。例如，主循环不断检查是否有数据到来。然而，在网络通信中，数据的到来是随机且异步的。为了高效处理网络事件，LwIP 采用了类似中断的思想：**你把一个处理函数的地址交给我，当事件发生时，我来调用这个函数。**

### 1.1 函数指针的定义与传递
在 C 语言中，函数名本身就是一个指向该函数代码段首地址的指针。
```c
// 1. 定义一个符合特定格式的函数
err_t my_recv_callback(void *arg, struct tcp_pcb *tpcb, struct pbuf *p, err_t err) {
    // 处理接收到的数据
    return ERR_OK;
}

// 2. 传递函数指针给底层注册接口
tcp_recv(tpcb, my_recv_callback); 
```

> [!TIP]
> 这里的 `my_recv_callback` 被作为参数传递给了 `tcp_recv` 函数。底层协议栈会将这个地址保存起来。这种被作为参数传递、并在特定事件发生时被第三方（底层库）调用的函数，就称为**回调函数**。

---

## 2. LwIP 中的回调函数实战

在 LwIP 的 **RAW API** 中，几乎所有的业务逻辑都是通过注册回调函数来完成的。

### 2.1 接收数据回调：tcp_recv()
当你建立一个 TCP 连接后，你不能用 `while(1)` 去死等数据。你需要告诉协议栈：“如果这个连接有数据来了，请调用我指定的函数”。
```c
// 注册接收回调
tcp_recv(pcb, http_recv_callback);
```
当网卡（MAC）收到数据包并层层解析，确认是发给这个 TCP 连接的负载数据后，LwIP 内部会自动触发调用 `http_recv_callback`。

### 2.2 连接接入回调：tcp_accept()
如果你写的是一个 TCP 服务端（Server），你需要监听某个端口。当有客户端尝试连接时，你需要一个回调来处理新连接。
```c
// 注册接受连接回调
tcp_accept(pcb, my_accept_callback);
```
在 `my_accept_callback` 中，你可以为这个新连接分配资源，并继续为它注册单独的 `tcp_recv` 回调。

---

## 3. 回调函数的上下文安全性

在编写回调函数时，必须时刻警惕其**执行上下文（Context）**：
1. **中断上下文还是线程上下文？** 
   如果在网卡的 RX 中断服务函数中直接调用 LwIP 的解析代码，那么你的回调函数也是在**中断中执行的**。此时回调函数内**绝对禁止使用任何阻塞操作（如延时、等待信号量）**，否则会导致整个系统崩溃。
2. **全局变量的安全性：**
   回调函数通常是异步执行的，如果回调函数修改了主循环也在使用的全局变量，必须加锁或使用 `volatile` 关键字防止编译器优化，这与中断函数的防错机制一致。
