# 结构体内存对齐与消除填充

在网络通信底层开发中，由于我们需要将一段连续的内存字节流“强转”为特定协议包的结构体进行解析，**结构体内存对齐（Memory Alignment）**和**字节填充（Padding）**成为了最容易踩坑的地方。

---

## 1. 什么是内存对齐与填充？

32 位单片机（如 ARM Cortex-M, RISC-V）的 CPU 在从内存读取数据时，通常希望数据的地址是 4 的整数倍（即 4 字节对齐）。如果数据不对齐，CPU 可能需要执行多次读取指令来拼接数据，导致性能下降，甚至在某些架构下会直接触发**硬件异常（HardFault）**。

因此，C 语言编译器（如 GCC）为了提升运行效率，会**自动在结构体的成员之间插入空字节（Padding）**，以保证下一个成员的地址能够对齐。

### 1.1 以太网 MAC 首部惨案
我们来看一个标准的以太网 MAC 层帧头（14 字节）：
```c
struct eth_hdr {
    uint8_t  dest[6];  // 目标 MAC 地址，6字节
    uint8_t  src[6];   // 源 MAC 地址，6字节
    uint16_t type;     // 协议类型，2字节
};
```
理论上 `6 + 6 + 2 = 14` 字节。
但如果我们使用 `sizeof(struct eth_hdr)` 打印大小，在某些编译器下，你可能会惊讶地发现它是 **16 字节**！
原因：编译器为了让 `eth_hdr` 整体能够被 4 字节对齐，在 `type` 后面自动填充了 2 个空字节。

> [!WARNING] 
> 在网络传输中，网线上发来的数据是严丝合缝的连续字节流。如果你用含有“填充字节”的结构体指针去映射接收缓存，**后续所有的字段都会发生错位**，解析出来的 IP 和端口将全是乱码！

---

## 2. 如何消除内存填充？

为了保证结构体与物理网络包 100% 映射，我们必须强迫编译器**取消内存对齐填充**，使其变为“紧凑型（Packed）”结构体。

### 2.1 GCC / Clang / MounRiver Studio 语法
使用 `__attribute__((packed))` 关键字：
```c
struct __attribute__((packed)) eth_hdr {
    uint8_t  dest[6];  
    uint8_t  src[6];   
    uint16_t type;     
};
```
此时 `sizeof(struct eth_hdr)` 将严格等于 14 字节。

### 2.2 Keil MDK (ARMCC) 语法
在老版本的 Keil 中，通常使用 `#pragma pack` 指令：
```c
#pragma pack(1)  // 设置 1 字节对齐（取消填充）
struct eth_hdr {
    uint8_t  dest[6];  
    uint8_t  src[6];   
    uint16_t type;     
};
#pragma pack()   // 恢复默认对齐方式
```

> [!TIP]
> 很多跨平台协议栈（如 LwIP）为了兼容不同的编译器，会定义一套包装宏，如 `PACK_STRUCT_BEGIN` 和 `PACK_STRUCT_END`，开发者只需将结构体包裹在这些宏之中即可保证可移植性。

## 3. 实战：数据包还原与强制类型转换

当我们通过以太网 DMA 接收到了一串数据缓存（存放于 `uint8_t buffer[1500]`），我们如何提取其中的数据？
利用紧凑型结构体配合**指针强转**（化零为整）：

```c
// buffer 开头的前 14 个字节必然是 MAC 头
struct eth_hdr *mac_header = (struct eth_hdr *)buffer;

// 判断协议类型是否为 IPv4 (0x0800)
// 注意：网络包是网络字节序，这里需要用到 ntohs 转换（详见大小端专题）
if (ntohs(mac_header->type) == 0x0800) {
    // 处理 IP 包
}
```
