# 代码仓库规范

> 版本：1.0.0  
> 更新日期：2026-06-21  

---

## 1. 仓库定位与架构

为了保持知识库的轻量与代码管理的独立性，本系统采用**双仓库隔离**架构：

- **主知识库** (`C:\Users\liyun\Documents\Embedded`)：存放 Obsidian 笔记、文档、图片等媒体资源。
- **子代码库** (`C:\Users\liyun\Documents\Embedded_Code`)：存放实际的代码工程、示例与演示源码。

### 目录链接关系
通过 Windows 目录联接（Directory Junction），子代码库被链接至主知识库的 `02代码库` 目录。
```
Embedded (主知识库)
├── 01待处理/          # 存放未整理的笔记与规范文件
└── 02代码库/          # 链接指向 Embedded_Code (子代码库)
      ├── 01待处理/    # 存放从外部迁移过来、尚未整理的代码工程
      └── (归类目录)/  # 整理完毕的代码工程目录
```

> [!NOTE]
> 主知识库的 `.gitignore` 已配置忽略 `/02代码库/`。主代码库的提交不会影响子代码库，反之亦然。两个仓库独立进行版本控制。

## 2. 代码生命周期管理规范

从外部项目、官方例程或其他地方迁移过来的代码，必须遵循以下生命周期：

### 2.1 待处理阶段（外部迁移）
1. 外部迁移的代码工程，统一放置在子代码库的 `02代码库/01待处理` 目录中。
2. 对应的学习笔记、参考资料放置在主知识库的 `01待处理` 目录中。
3. **禁止主动在子仓库中自行创建分类目录**。在没有收到明确的归档命令或整理指令前，所有过渡性代码必须且只能保留在 `02代码库/01待处理` 目录下。

### 2.2 归档整理阶段
1. 当主知识库中的笔记/归纳工作完成，且笔记确定了最终存放路径（例如：`Embedded/03单片机/STM32`）时。
2. 对应的代码工程应**同路径归档**。将代码从 `02代码库/01待处理` 迁移至 `02代码库/03单片机/STM32`（保持与笔记目录的一致性或对应关系）。
3. 归档后，在笔记中可以通过 Obsidian 相对链接引用代码目录中的源码文件。

---

## 3. 提交与变更纪律

1. 代码仓库的提交必须遵循项目的 `Git规范.md`。
2. 在进行任何跨目录的代码移动、归类整理前，必须保证当前分支处于 `clean` 状态（无未提交的更改），以便于随时回滚。
3. 迁移和整理代码时，尽量保持一次提交仅做一次目录调整，以便清晰追溯。
