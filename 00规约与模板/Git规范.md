## 1. 提交消息规范

为了保证 Obsidian 知识库的变更历史清晰、可追溯，所有的提交记录必须使用**中文**编写，并遵循结构化的提交格式。

### 1.1 消息结构

```
<type>(<scope>): <subject>

[可选 body]
```

### 1.2 type 类型说明

在笔记知识库的上下文中，请使用以下提交类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新增笔记 / 知识点 | 新增 DMA 控制器笔记、添加 HAL 库串口配置步骤 |
| `fix` | 订正笔记内容 | 纠正时钟树配置图参数、修改笔误与错别字 |
| `refactor` | 重构知识库结构 | 知识库目录结构优化、拆分或合并过长笔记 |
| `style` | 笔记排版与双链优化 | 格式化 Markdown 语法、修复无效的双向链接 |
| `docs` | 规范文件更新 | 更新 Git 规范、更新笔记排版规范 |
| `chore` | 配置文件与插件变更 | 升级 Obsidian 插件、调整 `.obsidian` 设置 |

### 1.3 scope 范围说明

范围用于标明本次修改的具体模块或知识领域，常见范围如下：

| 范围 | 说明 |
|------|------|
| `stm32` | STM32 相关的硬件与开发笔记 |
| `esp32` | ESP32/ESP-IDF 开发笔记 |
| `rules` | 知识库规则、模板、工作流规范 |
| `tool` | 编译器、烧录器或其它开发工具的使用笔记 |
| `todo` | 待处理（`01待处理`）目录下的草稿与整理工作 |

### 1.4 subject 简述要求
- 必须使用中文简述。
- 不超过 50 个字符。
- 不以句号结尾。
- 使用祈使语气（如“新增”、“修正”而非“新增了”、“修正了”）。

## 2. 完整提交示例

**示例 1：新增笔记**
```
feat(stm32): 新增 TIM 定时器中断配置笔记
```

**示例 2：修正错别字与技术错误（带正文）**
```
fix(stm32): 修正 DMA 竞态描述中的中断标志清除逻辑

变更内容：
- 修正原文中关于清除 DMA 通道传输完成标志的时序描述
- 补充在 CubeMX 中开启中断时的注意事项
```

**示例 3：笔记库重构**
```
refactor(todo): 将待处理的 I2C 笔记分类归档至单片机目录下
```

## 3. 提交纪律与最佳实践

### 3.1 提交粒度
- **单功能单提交**：一条提交只解决一个问题。例如：新增笔记和订正旧笔记应拆分为两个提交，严禁一次性提交数个不相关的变更。
- **可阅读性**：每次提交后，保证笔记的双链在当前状态下没有严重断联。

### 3.2 禁止事项
- **禁止使用模糊的提交描述**：如 `update`、`save`、`整理笔记` 等，这会导致提交历史失去追踪意义。
- **禁止在后台未经允许自动提交**：任何自动化的 Git 提交脚本需要受控使用，保证提交信息的规范性。
- **禁止将敏感信息提交**：如密码、私钥、个人凭证等。
