# 笔记规范

> 版本：2.0.0  
> 更新日期：2026-06-27  

---

## 1. 核心红线准则

为了防止知识库在前期开发和积累阶段因过度设计而导致文件夹冗余、杂乱，特制定以下**红线准则**：

> [!CAUTION]
> **未整理的资源全部放待处理目录！**  
> **没有整理命令或具体放置命令，严禁主动分配文件夹或创建新的分类目录！**

### 1.1 准则释义
- **“未整理的资源”**：包含从网页剪辑的草稿、临时记录的技术片段、未完成结构化整理的随手记、待归档的代码片段，以及**AI对话记录**、**文档教程**、**视频教程字幕**等参考资料。
- **“待处理目录”**：主知识库为 `01待处理` 目录，链接的代码子仓库为 `02代码库/01待处理` 目录。
- **“严禁主动分配文件夹”**：不允许根据主观猜测或参考“市面上的完美设计”提前建立大量空文件夹。有多少笔记就处理多少笔记，不追求绝对规范，内容积累到一定程度后，再逐步向规范靠拢。

## 2. 笔记生命周期

知识库的笔记流动应遵循**捕获-整理-归档**的极简流程：

```mermaid
graph TD
    A[外部输入/灵感/草稿] -->|统一放入| B(01待处理 目录)
    B -->|逐步细化/补充| C{内容是否整理完毕且确定归属?}
    C -- 否 -->|继续保留| B
    C -- 是且有明确整理命令 -->|移动归档| D(对应分类目录)
```

## 3. 待处理资源与产物组织规范

在 `01待处理` 目录中，资源与学习产物需按照以下目录树与规则进行管理：

### 3.1 主题划分
- **主题文件夹**：由开发者主动在 `01待处理` 下创建（例如：`01待处理/STM32`、`01待处理/FreeRTOS`），用以汇聚该主题下的所有待整理材料。

### 3.2 资源自动归类（Agent 职责）
当引入新的外部资源后，开发者会给出分类命令（例如：“整理资源到 STM32”）。Agent 必须根据资源类型，自动将资源划分到对应主题下的子文件夹中：
- `AI对话记录/`：保存与 AI 交流的原始对话记录（如 Markdown 或 text 格式的对话导出）。
- `文档教程/`：存放官方手册、开源库说明、PDF 章节或网页剪藏等文档。
- `视频教程字幕/`：存放视频教程的字幕提取文本、视频笔记草稿。

### 3.3 笔记产物放置规则
- **产物定位**：基于上述参考资源进行学习归纳后，产出的结构化笔记（Markdown 文件）。
- **放置规则**：**直接存放在该主题文件夹的根目录下**（例如：`01待处理/STM32/定时器双缓冲配置.md`）。不需要像参考资源那样单独分配到子文件夹中，以保证笔记处于易于预览和双链链接的扁平层级。

### 3.4 命名规则
- **核心概念与模块名称**：必须完全使用**清晰的中文自然语言**（例如：`定时器输入捕获测量频率.md`，而不是 `tim_input_capture.md`）。
- **专有名词**：特定芯片型号、协议名称、技术术语或专有名词保留英文大写或惯用写法（例如：`DMA数据传输机制.md`、`FreeRTOS信号量应用.md`）。

### 3.5 媒体附件存放规范
- 所有笔记引用的图片、PDF 等媒体附件，统一存放在主知识库根目录的 **99资源附件** 目录下（或该主题文件夹下的 **assets** 目录中）。
- 图片命名格式强制为：**笔记名称_序号.png/jpg**，禁止直接使用软件默认截图命名（如 **Pasted image xxx.png**）。

### 3.6 原始资源唯一命名法
- 导入至 **01待处理/主题/子目录** 的原始参考文件，必须按照以下格式重命名：
  - **主题名_类型代码_日期_描述.md**
  - 类型代码定义：**AI** 代表 AI 对话记录，**DOC** 代表文档教程，**SUB** 代表视频教程字幕。
  - 示例：**STM32_AI_20260621_串口DMA发送.md**

## 4. Obsidian 笔记构建规范

为了确保在 `01待处理` 目录中的笔记产物能够被 Obsidian 良好解析，并为未来一键归档做好准备，每一篇笔记产物在编写时必须遵循以下规范：

### 4.1 YAML Frontmatter (文档属性)
所有笔记产物顶部必须包含 YAML 元数据块，且必须包含以下核心字段：
```yaml
---
status: todo          # 状态：todo (待整理), processing (整理中), done (已归档)
created: 2026-06-21   # 创建日期，格式为 YYYY-MM-DD
tags:
  - stm32/timer       # 层级标签：主题/子领域
  - todo              # 状态标签
references:
  - "[[AI对话记录_20260621.md]]"  # 双链指向此笔记所依赖的原始参考资源
---
```

### 4.2 双链链接 (Double Links)

#### 4.2.1 链接语法速查

| 形式 | 语法 | 适用场景 |
|---|---|---|
| 笔记链接 | `[[笔记名称]]` | 关联整体主题 |
| 标题链接 | `[[笔记名称#标题]]` | 关联具体知识点章节 |
| 块链接 | `[[笔记名称#^blockid]]` | 关联原子知识点/单段代码 |
| 嵌入链接 | `![[笔记名称#标题]]` | 定义、公式、关键结论的复用 |
| 别名 | `[[笔记名称#标题|显示文字]]` | 让链接文字语义自洽 |

块 ID 手动命名规则：以字母开头、只含字母数字和连字符，格式为 `主题-子主题-关键词`(如 `^fsm-enter-exit`)。

#### 4.2.2 链接粒度选择原则

```
笔记级链接  →  关联同主题的"姊妹笔记"，作为拓展阅读入口
章节级链接  →  关联同一知识点在另一笔记中的对应章节
块级链接    →  关联原子知识点、定义、代码片段
```

**判断标准**：如果读者点击链接后需要滚动超过半屏才能找到相关内容,就必须换成更精确的标题链接或块链接。

#### 4.2.3 禁止泄漏上下文坐标系

**目标笔记的内部版本号、章节编号等坐标系,不得直接出现在引用方笔记的正文中作为读者可见的导航信号。**

- 禁止：在入门笔记中写"V3 查表法"——"V3"对入门读者毫无意义
- 正确：写"查表法",并链接到演进笔记的对应章节

#### 4.2.4 自包含原则

**每篇笔记的正文必须在不跳转链接的情况下能完整表达其定位范围内的概念。**链接只是"更多"的入口,不是理解正文的必要条件。

#### 4.2.5 双向链接约定

笔记 A 引用笔记 B 时,应同时在 B 中建立回链,确保图谱双向可达。

#### 4.2.6 上下文桥接写法

需要提示"更多细节在另一篇"时,采用标准桥接句,让句子在不跳转的情况下也能独立阅读:

```markdown
> [!TIP]
> 本节仅覆盖入门用法。若要了解 Enter/Exit 生命周期机制的完整实现,
> 参见 [[笔记名#具体章节标题|显示文字]]。
```

#### 4.2.7 关联笔记章节规范格式

笔记末尾的"关联笔记"区域,必须使用带章节/描述的格式,不得裸写笔记名:

```markdown
## 关联笔记

<!-- 错误：读者不知道关联点在哪 -->
- [[另一篇笔记]]

<!-- 正确：说明关联的知识点维度 -->
- [[另一篇笔记#具体章节|简短描述]] — 一句话说明点击后能得到什么
```

#### 4.2.8 YAML references 字段语义

`references` 字段记录笔记所**依赖的原始参考资源**(来源溯源),与正文中的知识关联双链不同。知识关联(姊妹笔记、进阶笔记)放在正文的"关联笔记"章节,不放在 references 里。

```yaml
references:
  # 仅放原始参考资源：AI 对话、文档、视频字幕
  - "[[原始对话记录文件名]]"
```

#### 4.2.9 资源追溯与知识网关联

- **资源追溯**：笔记中如果引用了 `01待处理` 中子目录下的原始参考资源,必须在 references 属性中建立双链。
- **知识网关联**：鼓励将笔记中的核心术语关联到已有的概念笔记上,例如：`利用 [[DMA数据传输机制]] 实现高效串口接收`。

#### 4.2.10 嵌入内容(Transclusion)规范

嵌入(`![[]]`)将目标内容直接渲染在当前页面,适合知识复用:

| 适合嵌入 | 不适合嵌入 |
|---|---|
| 复用定义/公式 | 超过 100 行的整篇笔记 |
| 复用代码片段(跨笔记引用同一段) | 与当前笔记主题无直接关系的内容 |
| MOC 汇总页面拼接各章节摘要 | 嵌入的内容本身已是引用(避免嵌套嵌入) |

#### 4.2.11 代码文件中引用笔记

代码文件不支持 Obsidian 双链,但可通过注释建立单向索引:

```c
/**
 * @file    module.c
 * @brief   模块简介
 * @note    对应知识库笔记：《笔记名》§章节
 *          路径：01待处理/主题/笔记名.md
 */
```

关键函数或"黑魔法"代码行上方可加行内索引:

```c
/* 基类首位原则。原理参见《笔记名》§4.2.1 */
```

#### 4.2.12 代码仓库链接

笔记引用代码库时使用**仓库内相对路径**:

```markdown
[模块实现](../../02代码库/01待处理/主题/模块/main.c)
```

GitHub Permalink 用于引用代码的具体实现(固定 commit 的行号链接),防止链接随代码变更失效。

> [!WARNING]
> 绝对路径(`file:///C:/...`)在多人共享或换机后会失效。统一使用仓库内相对路径或 GitHub Permalink。

#### 4.2.13 链接健康维护

- 笔记重命名时,Obsidian 会自动更新所有引用该文件的双链(需在设置中开启"自动更新内部链接")
- 块 ID 一旦被其他笔记引用,禁止删除或修改,只允许添加同义 ID
- 标题文字改动后,所有指向该标题的链接会变成死链,修改前需全局搜索并一并修复

### 4.3 标签设计 (Tag System)
- **层级标签**：统一使用 `/` 划分层级的标签，方便通过 Obsidian 标签树进行多维检索（例如：`#stm32/gpio`、`#rtos/task`）。
- **状态追踪**：待整理的笔记必须带有 `#todo` 标签。当笔记被整理归档时，需将 status 改为 `done`，并移除 `#todo` 标签。

## 5. 已消费资源的归档与清理机制

当笔记的状态（**status**）变更为 **done** 且笔记文件被移动至正式归档目录后，其所关联的原始参考资源按以下规则处置：
- **文档教程**：移动至全局的 **99归档/参考文档** 目录下，保持双向链接不失效。
- **AI 对话记录** 与 **视频教程字幕**：
  - 若无长期保存价值，可在确认笔记无误后直接删除（删除前需移除笔记中的对应双链，以防产生死链）。
  - 若需保留，则移动至全局 **99归档/对话记录** 目录。
