# 笔记规范

> 版本：1.1.0  
> 更新日期：2026-06-21  

---

## 1. 核心红线准则

为了防止知识库在前期开发和积累阶段因过度设计而导致文件夹冗余、杂乱，特制定以下**红线准则**：

> [!CAUTION]
> **未整理的资源全部放待处理目录！**  
> **没有整理命令或具体放置命令，严禁主动分配文件夹或创建新的分类目录！**

### 1.1 准则释义
- **“未整理的资源”**：包含从网页剪辑的草稿、临时记录的技术片段、未完成结构化整理的随手记、待归档的代码片段，以及**AI对话记录**、**文档教程**、**视频教程字幕**等参考资料。
- **“待处理目录”**：主知识库为 `01待处理` 目录，链接的代码子仓库为 `02代码库/01待处理` 目录。
- **“严禁主动分配文件夹”**：不允许根据主观猜测或参考“市面上的完美设计”提前建立大量空文件夹。有多少笔记就处理多少笔记，不追求绝对规范，内容积累到一定程度后，再逐步向规范靠拢。

## 2. 笔记生命周期

知识库的笔记流动应遵循**捕获-整理-归档**的极简流程：

```mermaid
graph TD
    A[外部输入/灵感/草稿] -->|统一放入| B(01待处理 目录)
    B -->|逐步细化/补充| C{内容是否整理完毕且确定归属?}
    C -- 否 -->|继续保留| B
    C -- 是且有明确整理命令 -->|移动归档| D(对应分类目录)
```

## 3. 待处理资源与产物组织规范

在 `01待处理` 目录中，资源与学习产物需按照以下目录树与规则进行管理：

### 3.1 主题划分
- **主题文件夹**：由开发者主动在 `01待处理` 下创建（例如：`01待处理/STM32`、`01待处理/FreeRTOS`），用以汇聚该主题下的所有待整理材料。

### 3.2 资源自动归类（Agent 职责）
当引入新的外部资源后，开发者会给出分类命令（例如：“整理资源到 STM32”）。Agent 必须根据资源类型，自动将资源划分到对应主题下的子文件夹中：
- `AI对话记录/`：保存与 AI 交流的原始对话记录（如 Markdown 或 text 格式的对话导出）。
- `文档教程/`：存放官方手册、开源库说明、PDF 章节或网页剪藏等文档。
- `视频教程字幕/`：存放视频教程的字幕提取文本、视频笔记草稿。

### 3.3 笔记产物放置规则
- **产物定位**：基于上述参考资源进行学习归纳后，产出的结构化笔记（Markdown 文件）。
- **放置规则**：**直接存放在该主题文件夹的根目录下**（例如：`01待处理/STM32/定时器双缓冲配置.md`）。不需要像参考资源那样单独分配到子文件夹中，以保证笔记处于易于预览和双链链接的扁平层级。

### 3.4 命名规则
- **核心概念与模块名称**：必须完全使用**清晰的中文自然语言**（例如：`定时器输入捕获测量频率.md`，而不是 `tim_input_capture.md`）。
- **专有名词**：特定芯片型号、协议名称、技术术语或专有名词保留英文大写或惯用写法（例如：`DMA数据传输机制.md`、`FreeRTOS信号量应用.md`）。

### 3.5 媒体附件存放规范
- 所有笔记引用的图片、PDF 等媒体附件，统一存放在主知识库根目录的 **99资源附件** 目录下（或该主题文件夹下的 **assets** 目录中）。
- 图片命名格式强制为：**笔记名称_序号.png/jpg**，禁止直接使用软件默认截图命名（如 **Pasted image xxx.png**）。

### 3.6 原始资源唯一命名法
- 导入至 **01待处理/主题/子目录** 的原始参考文件，必须按照以下格式重命名：
  - **主题名_类型代码_日期_描述.md**
  - 类型代码定义：**AI** 代表 AI 对话记录，**DOC** 代表文档教程，**SUB** 代表视频教程字幕。
  - 示例：**STM32_AI_20260621_串口DMA发送.md**

## 4. Obsidian 笔记构建规范

为了确保在 `01待处理` 目录中的笔记产物能够被 Obsidian 良好解析，并为未来一键归档做好准备，每一篇笔记产物在编写时必须遵循以下规范：

### 4.1 YAML Frontmatter (文档属性)
所有笔记产物顶部必须包含 YAML 元数据块，且必须包含以下核心字段：
```yaml
---
status: todo          # 状态：todo (待整理), processing (整理中), done (已归档)
created: 2026-06-21   # 创建日期，格式为 YYYY-MM-DD
tags:
  - stm32/timer       # 层级标签：主题/子领域
  - todo              # 状态标签
references:
  - "[[AI对话记录_20260621.md]]"  # 双链指向此笔记所依赖的原始参考资源
---
```

### 4.2 双链链接 (Double Links)
- **双向链接语法**：统一使用 Obsidian 的标准双链语法 `[[笔记名称]]`。
- **资源追溯**：笔记中如果引用了 `01待处理` 中子目录下的原始参考资源，必须在笔记开头或 references 属性中建立双链。
- **知识网关联**：鼓励将笔记中的核心术语关联到已有的概念笔记上，例如：`利用 [[DMA数据传输机制]] 实现高效串口接收`。

### 4.3 标签设计 (Tag System)
- **层级标签**：统一使用 `/` 划分层级的标签，方便通过 Obsidian 标签树进行多维检索（例如：`#stm32/gpio`、`#rtos/task`）。
- **状态追踪**：待整理的笔记必须带有 `#todo` 标签。当笔记被整理归档时，需将 status 改为 `done`，并移除 `#todo` 标签。

## 5. 已消费资源的归档与清理机制

当笔记的状态（**status**）变更为 **done** 且笔记文件被移动至正式归档目录后，其所关联的原始参考资源按以下规则处置：
- **文档教程**：移动至全局的 **99归档/参考文档** 目录下，保持双向链接不失效。
- **AI 对话记录** 与 **视频教程字幕**：
  - 若无长期保存价值，可在确认笔记无误后直接删除（删除前需移除笔记中的对应双链，以防产生死链）。
  - 若需保留，则移动至全局 **99归档/对话记录** 目录。
